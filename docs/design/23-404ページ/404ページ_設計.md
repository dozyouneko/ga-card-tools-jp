## 目的

存在しないパスが **404 ではなく「トップページを HTTP 200 で返す」**状態(soft 404)を解消する。

## 実測(2026-07-23・本番)

### 全領域で200が返る

| パス | 応答 | Content-Type |
|---|---|---|
| `/does-not-exist-12345` | **200** | text/html |
| `/cards/does-not-exist/` | **200** | text/html |
| `/tournaments/999999/` | **200** | text/html |
| `/tools/nope/` | **200** | text/html |
| `/api/nonexistent` | **200** | text/html |

返る本文は**トップページとバイト単位で完全に同一**(md5一致)。`<title>` も canonical もトップページのもの。

### 原因

リポジトリに **`404.html` が無い**(`_redirects` も無し、`pages_build_output_dir = "."`)ため、
Cloudflare Pages が未一致ルートに `index.html` を200でフォールバック配信している。

### 【2026-07-23 レビュー時に判明】SEOだけの問題ではない — 実際に画面が壊れている

**入れ子のパスでは、フォールバックされたトップページが完全に壊れて表示される。**
実装レビュー中に本番で実測した。

| パス | CSS適用 | カード描画 | status |
|---|---|---|---|
| `/does-not-exist`(1階層) | ✅ される | 50件 | `50 件を表示 / 全 2240 件` |
| `/cards/deleted-card-slug/` | ❌ **されない** | **0件** | **`読み込み中…` のまま** |
| `/tournaments/999999/` | ❌ **されない** | **0件** | **`読み込み中…` のまま** |

原因は `index.html` の CSS・JS 参照が**すべて相対パス**であること。
`/cards/xxx/` に対してフォールバックされると `style.css` は `/cards/xxx/style.css` に解決され、
**それもsoft 404で「200のHTML」が返る**ため、CSSともJSとも解釈されず無視される。
HTTPステータスは全て200なのでエラーとしても検知されない。

⚠️ **壊れるのは実際に起こりうるURL形状そのもの**である。
サイトマップに載せている **カード2,241件・大会421件** はいずれも入れ子のパスで、
スラッグ変更・大会の削除・URLの打ち間違いで**この壊れた画面に到達する**。

**したがって本issueは「SEOの衛生」ではなく「表示不具合の修正」である。**

### 影響

- **表示が壊れる**(上記。入れ子のパスでCSS・JSが効かない)
- **soft 404**: Googleがsoft 404と判定し、クロール効率とインデックス品質が下がる
  ([#22](https://github.com/dozyouneko/ga-card-tools-jp/issues/22)のクロール予算問題を悪化させる)
- **`/api/` の応答が壊れる**: JSONを期待するクライアントにHTMLが200で返る。
  robots.txt が `Disallow: /api/` のため検索面の実害は無いが、デバッグを難しくする
- **診断が難しい**: 今回のサイトマップ調査でも、URLを少し間違えると
  「XMLのはずがHTMLが200で返る」ため切り分けに時間がかかった

## ⚠️ 設計上の最重要点 — 既存ヘッダを流用してはいけない

`index.html` の `.site-header` には**ツールリンクの折りたたみ**がある。
[style.css:403-410](../../../style.css#L403-L410) のスマホ用メディアクエリ:

```css
.tools-toggle { display: block; ... }      /* スマホでのみ表示されるトグル */
.header-actions { display: none; }          /* ← リンクは既定で非表示 */
.site-header.tools-open .header-actions { display: flex; ... }   /* ← JSが .tools-open を付けて初めて出る */
```

`.tools-open` を付けるのは [app.js](../../../app.js) の `tools-toggle` ハンドラ**だけ**。
つまり **JSを読み込まない404ページでこのヘッダを流用すると、スマホではリンクが1つも表示されない**。
「行き先を案内する」という404ページの目的が完全に失われる。

**→ 折りたたみヘッダは流用せず、404ページ専用の「常に見えるリンク一覧」を本文に置く。**

## 設計

### A. `404.html` をリポジトリ直下に手書きで追加する

- **手書きにする**。ルート直下の手書きHTMLは現状 `index.html` のみで、
  `tools/*/index.html` も手書き。**生成スクリプトに組み込む必要はない**(データ依存が無いため)
- **JavaScriptを一切読み込まない**。
  - CSPが `script-src 'self'` のため**インラインスクリプトは使えない**
  - そもそも404ページが46本・1.6MBのJSを読むのは本末転倒([#22](https://github.com/dozyouneko/ga-card-tools-jp/issues/22))
- **`style.css` は流用する**(17KB)。`.site-header` / `.brand` / `.brand-mark` / `.site-footer` が揃っている

構成:

| 部分 | 内容 |
|---|---|
| ヘッダ | **ブランド表示のみ**(`.brand` + `.brand-mark` + サイト名)。**`.tools-toggle` と `.header-actions` は置かない** |
| 本文 | 見出し「ページが見つかりません」+ 簡単な説明 + **常時表示のリンク一覧**(下記) |
| フッタ | `index.html` の `.site-footer` と同じ免責文 |

本文のリンク一覧(すべて既存ページ):

- `/` カードDB(トップ)
- `/cards/` エキスパンション一覧
- `/tournaments/` 大会デッキ検索
- `/tools/deck-builder/` デッキ構築ツール
- `/tools/glossary/` ひとくちキーワード解説
- `/tools/print/` 印刷用PDF作成

### B. CSS は `style.css` に最小限のクラスを足す

404本文用のクラス(例 `.notfound`)を `style.css` の末尾に追加する。
**メディアクエリの中には入れない**(常時表示が要件のため)。

⚠️ **`style.css` は他の全ページが読み込む共有ファイル**。既存クラスの定義は変更せず、
**新規クラスの追加のみ**にとどめること(検証項目8で回帰を確認する)。

### C. `<meta name="robots" content="noindex">` を付ける

404ステータスが正しく返るなら noindex は不要だが、**`/404.html` というURL自体は
200で直接アクセスできてしまう**ため、そのURLがインデックスされないように付けておく。

### D. `_headers` は変更しない

`/*` のルール(CSP・`X-Content-Type-Options` 等)がそのまま適用される。
外部リソースを一切使わないため追加の許可は不要。**検証項目9で違反が出ないことを確認する。**

### E. sitemap には入らない(確認済み)

[scripts/build-card-pages.mjs](../../../scripts/build-card-pages.mjs) の `buildSitemap()` は
**`add(...)` の明示リスト方式**でファイルを走査していない。
`404.html` を置いてもsitemapに混入しない(検証項目7で確認)。

## 実装ステップ

1. **`404.html`** をリポジトリ直下に作成(ヘッダはブランドのみ・本文にリンク一覧・JSなし・noindex)
2. **`style.css`** の末尾に `.notfound` 系のクラスを追加(既存定義は触らない)
3. 動作確認(下記の⚠️に注意)

⚠️ **再ビルドは不要**。sitemapもカード・大会ページも影響を受けない。

## ⚠️ 検証の制約 — ローカルでは本質的な確認ができない

**`npm run dev`(scripts/serve.mjs)は Cloudflare Pages のフォールバック挙動を再現しない。**
`_headers` も解釈しない(CLAUDE.md)。

そのため:

| 確認できること | 場所 |

| **未定義パスが404ステータスを返すこと** | **本番のみ**(push後の `curl -sI`) |

**この issue は「push後に本物で確認するまで完了しない」**。実装報告の時点では
ローカル確認ぶんのみを報告し、P項目は公開後に実施すること。

### 検証項目14の内訳(回帰なし)

- **主要ルート14本すべて200**: `/` `/cards/` `/cards/<slug>/` `/tournaments/` `/tournaments/<id>/`
  `/tools/deck-builder/` `/tools/glossary/` `/tools/print/` `/sitemap.xml` `/robots.txt`
  `/style.css` `/app.js` `/shared/vendor/jspdf.umd.min.js`
- **サイトマップ掲載URLから無作為12本を抽出 → すべて200**(カード・大会・セットの各種)
- **既存API**: `/api/health` → **200**(JSON)、`/api/me` → **401 `{"error":"not_logged_in"}`**(未ログイン時の正しい応答)

### ⚠️ `/sets/` が404になるのは回帰ではない

確認リストに入れていた `/sets/` は404になるが、**これは仕様どおり**である。

- `sets/index.html` は**存在しない**(56個のサブディレクトリのみ)
- **サイトマップにも入っていない**(掲載は `/sets/<slug>/` の56件のみ。いずれも200を確認)
- **サイト内から `/sets/` へのリンクは0件**

つまり元から実体の無いパスで、**これまで soft 404 で200を返していたものが正しく404になった**だけ。
設計時の確認リストに `/sets/` を「200であること」として載せたのは**設計側の誤り**。

### ⚠️ 検証項目15の期待値を訂正 — `/404.html` は308になる

```text
/404.html → 308  (Location: /404)
/404      → 200  （noindex あり）
```

CLAUDE.mdに記録のある**Cloudflare Pagesの拡張子トリム**が効くため、`/404.html` は
`/404` へ308リダイレクトされる。**設計に書いた「200であること」は誤りで、正しくは308→200。**
ページ自体には到達できるので問題はない。

`/404` が200で直接開けてしまう点は**設計C(noindex)が想定どおり効いており**、
`<meta name="robots" content="noindex">` が入っていることを実測で確認した。

  優先順位に依存する。**検証項目13で必ず実測すること。** 404にならない場合は
  `functions/api/[[path]].js` でJSONの404を返す案を別途検討する(本issueのスコープ外)
