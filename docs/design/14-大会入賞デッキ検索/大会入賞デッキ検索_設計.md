## 目的・差別化ポイント

大会(Regionals/Store Championship等)の入賞デッキを日本語訳付きで検索・閲覧できるようにする。**大会入賞結果を日本語訳付きで見られるサイトは現時点で他に存在しない**(2026-07-19時点調査)。カードDBの既存翻訳データをそのまま適用できるのは本サイトならではの強み。

参考事例: カードゲームショップCANTRIPのデッキ検索 <https://cantripmtg.com/pages/decks-grand-archive>(手動キュレーションによる自社ページ。出典表記なし)

### 決定事項(2026-07-19ユーザー決定)

1. **対象大会**: **主要大会のみ・全世界**。カテゴリ許可リスト = **`store-championships` / `regionals` / `nationals` / `ascent`**(2026-07-20拡大。実データで存在確認済み。`invitational`/`worlds`は今回のスキャン範囲では未検出のため引き続き未追加・判明次第追加)。`regular` は除外。また`status`が`canceled`のイベントは許可リストに一致してもその場で除外する(pendingにも入れない。テストイベント等が永久にpendingへ残留するのを防ぐ)
2. **実装方針**: **静的生成**(`scripts/build-card-pages.mjs` と同方式)。ビルド時にAPIをスキャンし、対象大会データをgit管理のJSONに保存→静的HTMLを事前生成。Cloudflare側の追加費用・サーバー処理なし。既存の「翻訳データはgit管理→pushで自動デプロイ」の運用にそのまま乗る

### 既存資産の再利用(カード名→slug解決)

デッキ構築ツールの「Omnidexインポート」機能([tools/deck-builder/app.js:1671-1714](../../../tools/deck-builder/app.js#L1671-L1714))が、まさに同じ課題(Omnidex提出フォーマットの英語カード名→slug解決)を解決済み:

- `normCardName()`: 記号・大小文字ゆれを吸収する正規化(`toLowerCase().replace(/[^a-z0-9]+/g,"")`)
- 完全一致のみ採用(曖昧一致は誤登録防止のため不採用)という方針も踏襲する

本機能はビルド時のバッチ処理のため、ライブAPI呼び出し(`resolveCardSlug`)ではなく**ローカルのカードスナップショット(`tmp/api-cache/cards-snapshot.json`)を使った同ロジックの一括解決**に置き換える(`scripts/build-card-pages.mjs` が同スナップショットを既に利用しているのと同じ考え方)。

### 読み取り専用デッキ表示との構造共通性

大会デッキは「編集不可」という点でデッキ構築ツールの共有画面(`#deck/<id>`)の表示ロジックに近い。データを `{card_slug, board(material/main/side), qty}` の形に正規化しておけば、統計タブの `computeDeckStats()` 等の集計ロジックを**そのまま入賞デッキにも適用できる可能性がある**(例: 「この大会の使用エレメント分布」)。ただし今回のMVPスコープでは表示のみとし、統計流用は将来検討(#3のコストカーブ検討と合わせて考える余地あり)。

## 運用リスク: ファイル数上限・リポジトリ容量(2026-07-20調査)

### Cloudflare Pagesのファイル数上限(近い将来に深刻化しうる・最優先で対処)

Cloudflare Pagesは1デプロイあたりのファイル数上限が **Freeプラン=20,000件・有料プラン=100,000件**(2026-01の変更で有料プランは20,000→100,000に緩和。[Cloudflare changelog](https://developers.cloudflare.com/changelog/post/2026-01-23-pages-file-limit-increase/))。

| | ファイル数 |
|---|---|
| 導入前(既存トラッキング済みファイル) | 2,418(カード2,241件+その他) |
| 今回の追加分(大会421+デッキ10,960+一覧1。**1デッキ=1ファイル**の3階層構成の場合) | 約11,382 |
| **導入後の合計** | **約13,800** |

Freeプランなら1シーズン分のバックフィルだけで上限20,000の約69%を消費する計算になる。RDOシーズン(約4ヶ月)ごとに同規模の増分が続くため、**次シーズンが終わる頃にはFreeプランの上限を超え、サイト全体のデプロイが失敗し始める**リスクが高い(Cloudflare Pagesはファイル数超過時にビルド自体を失敗させるため、カード・機能の更新も含めて全て止まる)。有料プランなら100,000件まで余裕があるが、本プロジェクトのプラン種別は`wrangler.toml`等のリポジトリ内設定からは判別できず未確認。

**ユーザー決定(2026-07-20): ページ構成を2階層(デッキを大会詳細ページに内包)に戻す。** 1デッキ=1ファイルをやめることで、ファイル数の基準が「デッキ数」(約1万・増え続ける)から「大会数」(約420・同ペースでも遥かに緩やか)に変わり、約96%削減できる。詳細は「静的ページ」節を参照。

### GitHubリポジトリの容量(2026-07-21訂正: 懸念は解消)

**前版までの記述(「221〜300MB規模になり`.git`が10〜40倍に跳ね上がる」)は前提が誤っていた。** `.github/workflows/build-tournaments.yml`のcronを無効化した理由(「初回実行でmainへ約266MBがpushされ、履歴からの取り消しが困難」)も同じ誤りに基づく。

誤りの内容: 221〜266MBは**作業ツリー上のバイト数**であり、gitはオブジェクトをzlib圧縮して保存する。生成HTMLはカードタイルの繰り返しで冗長性が極めて高く、実測すると圧縮率は11分の1だった。

| 項目 | 実測値(2026-07-21) |
|---|---|
| 現`tournaments/`(74大会分)の作業ツリー | 33MB |
| 同 gzip -9 圧縮後 | **2.9MB(11分の1)** |
| 420大会フル生成時の作業ツリー見込み | 約126MB(ハイブリッド方式なら約93MB) |
| **同 リポジトリ実増加の見込み** | **約10〜15MB** |

現在の`.git`は6.9MB。初回バックフィルで20MB前後になる計算で、GitHubの推奨(1GB未満)・警告ライン(5GB)に対して十分に小さい。日次の自動更新は**新規に確定した大会の分しか差分が出ない**ため、以降の増加ペースはさらに緩やか。

残る注意点は「生成スクリプトを変更して全420ページを作り直すたびに、同規模(10〜15MB)の履歴が積み増される」こと。本タスクでは既に3回の方針変更で再生成が発生しているため、**方針を確定させてから一括で再生成・コミットする**運用とする(本改版でハイブリッド方式に確定させたのはこのため)。

**結論: 容量はcron有効化の阻害要因ではない。** 本改版の反映・実機確認が済んだ時点で`build-tournaments.yml`のcronコメントアウトを解除してよい(実装ステップ7)。

### 保留イベントの扱い(`data/tournaments/pending.json`)

イベントIDは**大会開催前に先行作成される**(実測: startAtが未来の日程で登録済みのIDが多数存在。例: 現在の最大IDより前に2026-07-25/07-28開催のイベントが既に存在)。そのため「idを見つけた時点でカテゴリ対象=即確定」とはできず、以下の状態管理が必要:

```json
// data/tournaments/pending.json
[
  { "id": 58520, "firstSeenAt": "2026-07-19", "category": "store-championships", "startAt": "2026-07-25T..." }
]
```

- 新規スキャンでカテゴリ対象と分かったが `status !== "complete"` または `decklists` が空のイベントは、`events/<id>.json` を確定させず `pending.json` に記録
- 毎回のスキャンで **pending内の全idを再チェック**(件数は少数に留まる想定)。`status === "complete"` かつ decklists が揃っていれば `events/<id>.json` を確定生成し `pending.json` から除去
- `firstSeenAt` から一定期間(例: 60日)経っても確定しない場合は「開催中止・記録なし」とみなし `pending.json` から削除(無限に溜まるのを防ぐ)
- **(2026-07-20追加)** `status === "canceled"` が判明した時点で60日待たずに即座に`pending.json`から削除する(実測でid 50271「Test」のような`canceled`のテストイベントを確認済み。永久にpendingへ残留させない)

### 自動反映(GitHub Actions・2026-07-19決定)

リポジトリはpublicのため、GitHub Actionsの実行時間は無料枠に実質制限がない。以下の運用で新規大会を自動反映する:

```yaml
# .github/workflows/build-tournaments.yml (イメージ)
on:
  schedule:
    - cron: "0 18 * * *"   # 毎日 03:00 JST(イベント作成頻度は約115件/日・世界全体なので日次で十分)
  workflow_dispatch: {}     # 手動実行(動作確認・不具合対応用)
permissions:
  contents: write
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: node scripts/build-tournament-pages.mjs
      - run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add data/tournaments tournaments sitemap.xml
          git diff --cached --quiet || git commit -m "大会データ自動更新 $(date -u +%Y-%m-%d)"
          git push
```

- **mainへ直接push**(PRを経由しない)。理由: 追加専用(既存イベントは確定後書き換えない)の公開データであり、秘密情報・設定変更を含まないため、レビューを挟まなくてもリスクは低いと判断(ユーザー決定)
- pushをトリガーに既存のCloudflare Pages自動デプロイがそのまま動く(新しいデプロイ経路は不要)
- 実行ログはGitHub Actionsの実行履歴で確認できる(異常があれば`workflow_dispatch`で手動再実行して調査)
- スキーマ変更等でスクリプトが失敗した場合はワークフローが失敗するのみ(pushされない)。次回改修まで大会データは古いまま=安全側に倒れる

#### (2026-07-21確定 → **2026-07-22廃止**)巨大大会のみデッキを後読みする「ハイブリッド方式」

> ⚠️ **この節は2026-07-22の改訂で廃止されました。** デッキをダイアログ表示に変更したため、内包/後読みの分岐(`INLINE_DECK_LIMIT`)は不要になりました。**現行仕様は「ローカル確認で判明した改善点(2026-07-22)」節のA項を参照してください。** 以下は判断の経緯として残します(閾値100が体感上問題なかったという実測結果は、ダイアログ方式の妥当性の裏付けにもなっています)。

全デッキを内包すると、**デッキ数が突出して多い一部の大会だけ**ページが肥大して利用者の体感を損なう。実測した分布は以下のとおりで、影響を受けるのは420大会中14大会(3%)に限られる。

| 大会規模 | 該当数 | ページを開いたときの体感(A案=全内包の場合) |
|---|---|---|
| 中央値(15デッキ) | — | 180KB(転送36KB)・体感即時 |
| 上位10%(45デッキ) | 42大会 | 0.5MB(転送100KB)・ほぼ即時 |
| 100デッキ超 | **14大会** | 1.2MB〜・スマホで一瞬もたつく |
| 最大(422デッキ・id 51918 Fayble Icebound Slam Cup) | 1大会 | **4.8MB(転送約1MB)/カード画像タグ約25,000個** → スマホで数秒待ち・スクロールがカクつく |

**方針: 閾値`INLINE_DECK_LIMIT = 100`で分岐する。**

- **デッキ数 ≦ 100(406大会・97%)**: 従来どおり `deckAccordion()` の出力を `index.html` に内包する。**展開時の通信ゼロ・ページ内検索(Ctrl+F)で全デッキ横断・デッキ内容がSEOインデックス対象**という利点をそのまま維持する
- **デッキ数 > 100(14大会・3%)**: `index.html` には各デッキの `<details>` の **summary(順位・プレイヤー名・戦績)だけ**を出力し、中身は空にする。デッキ本体は同じ `deckAccordion()` が生成するHTMLを全デッキ分連結した断片 `decks.html` として別ファイルに書き出し、展開時にfetchして差し込む

**この方式を選ぶ理由**: 断片ファイルは**サーバ側の `deckAccordion()` の出力そのもの**なので、JS側にカード描画ロジックを二重実装しなくて済む(fetchして `innerHTML` するだけ)。JSONを取得してクライアントで組み立て直す案(B案)は、描画ロジックが二重化するうえページ内検索・SEOを全大会で失うため不採用とした。

実装の詳細:

- `decks.html` は完全なHTMLドキュメントではなく**断片**(`<html>`等を含めない)。`<details class="deck-acc" id="deck-...">` の連結のみ
- 差し込みは初回展開時に1回だけ行い、以降はDOMに残す(2回目以降の展開は即時)。**ページ読み込み後に`requestIdleCallback`でプリフェッチしておくと初回展開も体感ゼロにできる**(実装推奨)
- 読み込み中は `<details>` 内に「読み込み中…」を表示。失敗時は再試行リンクを出す(ネットワーク断で無反応にならないこと)
- 順位表の「デッキを見る」(`#deck-<player>`)は、**内包・後読みどちらでもリンク先の`<details>`自体は存在する**ため導線は変わらない。後読みの大会では「取得完了 → 展開 → スクロール」の順で処理する。URLハッシュ付きで直接開かれた場合(共有リンク)も同じ経路を通す
- CSPへの影響なし: 断片HTMLはstyle属性を含まず(`style-src 'self'`に抵触しない)、fetch先は同一オリジン
- `sitemap.xml` には `decks.html` を**含めない**(断片であり単独のページではないため)
- 生成ファイル数への影響は+14件のみ(合計約2,850件。Freeプラン上限20,000件に対して14%)

- 個別カードは `cards/<slug>/` への通常リンク。可能ならカードDBと同じ詳細モーダル(`shared/js/card-detail.js` + `card-i18n.js`)を読み込み、その場でポップアップ表示できるようにする(既存ページと同じUX)
- `sitemap.xml` への合流: `scripts/build-card-pages.mjs` のsitemap生成ステップが `tournaments/*/index.html` も走査して追加できるよう改修する(実行順序: 大会スキャン→カードページ生成、の順に実行する運用とする)
- **(2026-07-20確定)** サイト全体のヘッダnav(`index.html`・`cards/<slug>/`等の共通ヘッダ)に「大会入賞デッキ」リンクを追加する。既存カードページ2,240件の再生成を伴うが、`npm run build:cards` の通常再生成に含めれば追加コストなし。具体的な追加箇所は2つ:
  - [index.html:41-46](../../../index.html#L41-L46) の `#header-actions`(`ツール ▾`トグル配下)に他の`tool-link`と同じ形式で追加。モバイルは既存のJSトグル([style.css:300](../../../style.css#L300)の`.tools-open .tool-link`)にそのまま乗るため追加対応不要
  - `scripts/build-card-pages.mjs:191`の`siteHeader()`関数(`cards/<slug>/`等の全生成ページ共通ヘッダ)。`scripts/build-tournament-pages.mjs:372`の同名関数には**追加済み**なので、そちらの文言と揃える。`.cp-site`は`flex-wrap`済みでモバイルでも折り返し表示になる([build-card-pages.mjs:458](../../../scripts/build-card-pages.mjs#L458))

## ローカル動作確認で判明した改善点(2026-07-20)

ユーザーがローカルサーバーで生成済みページを実際に操作して発見した3点。

### 1. 国名表示バグ(「jpJP」のように見える)

`scripts/build-tournament-pages.mjs`の`flag(cc)`(325-328行目)は国コードをUnicode地域指示記号2文字に変換してflag絵文字にする実装だが、**絵文字非対応の環境ではただの文字っぽいグリフにフォールバックする**。さらに一覧ページ(400行目)・大会詳細ページ(497行目)で`flag(cc) + cc`を**セパレータなしで連結**しているため、絵文字が描画されない環境では国コードが二重に見えてしまう(例: 🇯🇵JP → 環境によっては「JPJP」寄りの表示)。`<select id="f-country">`(425行目)はスペース区切りのため症状は軽微。

**対応**: 400行目・497行目の連結箇所に区切り(スペースや中点)を入れる。絵文字が描画されない環境でも国コードが重複して見えないようにする。

### 2. 一覧テーブルに「開催国」カラムを追加

現状、国データは各行に`data-country`属性(396行目)として保持されているが、表示は`.ev-host`内にホスト名と一緒に埋め込まれるのみ(400行目)で独立した列がない。ヘッダ(434行目)に`<th>開催国</th>`、行テンプレート(400行目付近)に対応する`<td>`を追加する(バグ1の表記修正と合わせて実装)。

### 3. シーズンでの絞り込みを追加

現状のデータモデルには`season`フィールドが存在せず、`startAt`の日時のみ保持している。シーズン名(RDO/PTM等)は大会名の自由文字列に紛れ込んでいるだけで構造化されていない。

**対応方針**: スキャン時(`collectEvent`)またはインデックス生成時に、`startAt`と「シーズン期間(上記「シーズン期間」表を参照。`omni.gatcg.com/api/season?id=N`で正確な開始・終了日時を取得可能)」を突き合わせて`season`(例: `"RDO"`)を導出し`data/tournaments/index.json`・`events/<id>.json`に保持する。一覧ページの絞り込み用に`data-season`属性を各行に追加し、既存のクライアント側フィルタ(`LIST_JS`、689-734行目の`apply()`)に他の`<select>`と同じパターンで`#f-season`を追加する。シーズン境界はハードコードの定数テーブルとして管理し、新シーズン開始時に手動更新する運用とする(API側にシーズン一覧の取得エンドポイントが見当たらないため)

## ローカル確認で判明した改善点(2026-07-22)

ユーザーがローカルサーバーで全420大会の生成結果を実機確認した結果([検証/ローカルサーバー確認結果.md](検証/ローカルサーバー確認結果.md))。**U3〜U5(巨大大会の体感・展開待ち・カード詳細ダイアログ)は問題なし**で、`INLINE_DECK_LIMIT = 100` の妥当性は確認できた。以下は残る改善要望と、調査で判明した事実の訂正。

### A. デッキ表示をダイアログ化し、内包/後読みの分岐を廃止する【最重要】

**背景となる訂正: Ctrl+Fによる全デッキ横断検索は最初から成立していなかった。** カード名は属性にしか存在せずテキストノードが無いため、ブラウザの検索が届かない:

```html
<div class="cardph" data-slug="spirit-of-wind"><img alt="Spirit of Wind（風の精霊）" title="..." ...>
```

2026-07-21版はこの機能を「内包方式の最大の利点」としてB案(クライアント側描画)を却下する根拠にしていたが、**その根拠は事実誤認だった**。残る内包の利点はデッキ内容のSEOのみとなり、**ユーザー決定によりこれは手放してよい**(集客主軸はカードDBページであるため)。

**新方式:**

```text
tournaments/
├── index.html            … 大会一覧
└── <event-id>/
    ├── index.html         … 大会詳細: 概要 + 順位表のみ(デッキ一覧は持たない)
    └── decks.html         … 【全大会で生成】デッキ部分のHTML断片
```

- **`INLINE_DECK_LIMIT` と内包/後読みの分岐を廃止**し、`decks.html` を**全420大会で生成**する。大会詳細ページ本体は「概要+順位表」のみとなり、全大会で軽量になる(現状の内包側最大1.5MB→数十KB規模)
- 順位表の「デッキを見る」→ **ダイアログで該当デッキ1件を表示**する。デッキのアコーディオン一覧は廃止(ユーザー要望「ダイアログで表示できるのであれば提出デッキの一覧は不要」)
- **`decks.html` は取得後もテキストのまま保持し、`DOMParser` で解析して該当デッキ1件だけをダイアログに描画する。** 断片全体をDOMへ差し込まないこと(422デッキ分=カードタイル約25,000個をDOMに載せない。現行のアコーディオン方式より軽くなる)
- プリフェッチは現行同様 `requestIdleCallback` で取得のみ先に済ませる
- **カード詳細ダイアログとの入れ子**: デッキダイアログ内のカードをクリックすると既存の `shared/js/card-detail.js` のダイアログが上に重なる。Escapeは**内側(カード詳細)→外側(デッキ)の順**で閉じること。スマホ幅(≤640px)での全画面シート化は `card-detail.js` と同じ挙動に揃える
- **`#deck-<player>` のURLは維持する**(共有・ブラウザバック)。ダイアログを開いたらhistoryにハッシュを付与し、閉じたら除去する。ハッシュ付きURLで直接開かれた場合はダイアログを復元する
- 取得失敗時はダイアログ内に再試行導線を出す(現行の実装を踏襲)
- `deckAccordion()` の出力をそのまま使う原則は維持(クライアント側にカード描画ロジックを二重実装しない)

### B. 順位表を100行ごとに折りたたむ

**ユーザー要望**: 「大規模大会だと縦に画面が長すぎる。100行ごとで折りたたみにできないか。**できれば一つが開いているときは他の折りたたみが自動で閉じる**ようにしたい」「**スマホで見る場合が特に見づらい**ので順位表は折りたたみにしてほしい」。

- **101名以上の大会のみ**(実測14件・最大422名)に適用する。100名以下はこれまでどおり素の表を表示する
- `<details>` で「1〜100位」「101〜200位」…のブロックに区切り、**最初のブロックのみ開いた状態**にする
- **排他制御**: あるブロックを開いたら他のブロックを閉じる
- `#deck-<player>` で直接開かれた場合は、**該当プレイヤーを含むブロックを開いてから**ダイアログを表示する

### C. 国名を日本語表示にする(国旗絵文字を廃止)

**原因の訂正**: 2026-07-20版の「絵文字が出ない環境で重なって見えるため空白で区切る」という対処は**原因に届いていなかった**。真の原因は**Windowsが国旗絵文字のグリフを持たない**こと(Segoe UI Emojiに国旗は非収録)で、Unicode地域指示記号は非対応環境で**「JP」という2文字にフォールバック**する。したがって `🇯🇵 JP` は「JP JP」と表示され、空白区切りでは解消しない。

- **`flag()` による国旗絵文字を廃止**し、**日本語の国名**を表示する(例: `日本` / `アメリカ合衆国` / `台湾`)。日本語訳サイトという性格にも合致する
- 対応表は定数で持つ。**収録実績のある31カ国**は次のとおり(実測): `AE AU BE BN CA CL CZ DE DK ES GB HK HR HU ID IT JP KR KW LB MY NL NO NZ PH PL PR SG SI TW US`
- 未知の国コードは**コードをそのまま表示**してフォールバックする(新規国が増えても壊れない)
- 適用箇所は**3箇所すべて**: 一覧の「開催国」列 / 一覧の絞り込み `<select id="f-country">` / 大会詳細の概要

### D. 戦績に不戦勝(bye)を反映する — 順位付け自体は正しい

**調査結果: 不具合ではなかった。** ユーザー指摘の「RDO Store Championship　in 昭和町店(id 57553)の戦績と順位がおかしい」を、APIの生レスポンスまで遡って検証した。

`GET /omnidex/events/{id}/standings` の配列順は **`statsScore`(勝ち点)の降順に完全一致**しており(31名の大会で違反0件)、`rank = 配列順` とする現行実装は正しい。疑義の原因は**不戦勝(bye)が表示されていない**ことだった:

| 順位 | 現在の表示 | 実際の勝ち点 | 内訳 |
|---|---|---|---|
| 1位 | 1-0-1 | **7** | 1勝 + **bye 1** + 1分 |
| 2位 | 2-0-1 | **7** | 2勝 + 1分 |

- APIは `statsScore`(勝ち点)・`statsByes`(不戦勝数)を返しているが、**現行の`collectEvent()`はこの2つを保存していない**。両方を `events/<id>.json` に保存する
- 順位表に**「勝ち点」列を追加**する。byeがある選手は成績欄に併記する(例: `2-0-1（不戦勝1）`)
- 脚注を「**順位は勝ち点順、同点はタイブレーカーによる**」と明記する
- ⚠️ **この項目はデータの再取得を伴う**ため `--no-scan` では対応できない。全420大会の再スキャンが必要(下記「実装ステップ」参照)

### E. スマホ幅で順位表の右が見切れる

順位表は `.cp-scroll`(`overflow-x:auto`)で横スクロール自体は可能だが、**一覧表にある「低優先列を隠す」`@media (max-width:640px)` が順位表には入っていない**(一覧表は `#ev-table` の4列目を隠している)。同じ手法で順位表の **GW%・MW% 列を640px以下で非表示**にする。

### J. 不戦勝を勝ち数に含めて表記する

現在の `1-0-1（不戦勝1）` を **`2-0-1（うち不戦勝1）`** の形式にする(ユーザー指定)。

APIの `statsWins` は不戦勝を含まないが、`statsScore`(勝ち点)は**不戦勝を勝ちとして計算**している(1勝+bye1+1分=7点)。表示の勝ち数にbyeを加算することで勝ち点と辻褄が合い、順位が自然に読めるようになる。

- 表示する勝ち数 = **`statsWins + statsByes`**
- 注記は `（うち不戦勝N）`。**`statsByes === 0` の場合は注記そのものを出さない**
- 実例(id 57553): 1位 = `2-0-1（うち不戦勝1）`・勝ち点7 / 2位 = `2-0-1`・勝ち点7 → 同点であることが表記から読み取れる

### K. パンくずリストから戻ったときも検索条件を維持する

I項でURLクエリによる復元を入れたが、**ブラウザバックでは維持される一方、大会詳細のパンくずリンクから戻ると条件が失われる**(パンくずが素の `/tournaments/` を指しているため)。

- **一覧の行クリックで大会詳細へ遷移する際、現在のフィルタのクエリを大会詳細のURLに引き継ぐ**
- **大会詳細のパンくずの一覧リンクは、そのクエリを付け直した `/tournaments/?...` を指す**
- これによりブラウザバック・パンくずのどちらでも同じ絞り込み状態に戻る。ヘッダnavの「大会デッキ」リンクは**素の一覧**(条件なし)のままでよい

### L. 「入賞」の表記を削除する

- **トップページのボタンのみ**「🏆 **大会デッキ検索**」にする
- それ以外の **`大会入賞デッキ` はすべて `大会デッキ`** にする(「入賞」を削除)
- 対象は文言が出るすべての箇所: 共通ヘッダnav 2種(`build-card-pages.mjs`・`build-tournament-pages.mjs` の `siteHeader()`)・パンくず・`<title>`・meta description・OGP・見出し
- ⚠️ ヘッダ文言の変更のため**カードページ2,240件+セット56件の再生成が再度必要**(`npm run build:cards`)
- **対象はサイト利用者に見える文言のみ。** 設計フォルダ名(`14-大会入賞デッキ検索/`)・設計書ファイル名・過去のコミットメッセージ・issueタイトルは**変更しない**(リンクが壊れるため。CLAUDE.mdの「過去の設計フォルダの番号は変更しない」と同じ方針)

### N. スマホで検索条件エリアを折りたたむ

一覧ページの絞り込み入力群が、スマホ幅では縦に長すぎて一覧本体が見えない。

- **640px以下でフィルタ群を `<details>` に格納し、既定は閉じた状態**にする(PC幅では従来どおり常時表示)
- トグルの見出しは「絞り込み」。**適用中の条件数をバッジ表示**する(例: `絞り込み 2`)と、閉じていても状態が分かる
- 期間指定(F項)を含む全フィルタをこの中に入れる

## ローカル確認・第3回で判明した改善点(2026-07-22・O)

J〜N実装後のユーザー確認(スマホ幅の実機スクリーンショット)で、**テーブル表示がスマホ幅では破綻していること**が判明した。

### O. スマホ幅ではテーブルをやめてカード表示にする

**現状の問題**(実機スクリーンショットより):

| 画面 | 症状 |
|---|---|
| 一覧 | 大会名が3行に折り返し、開催国が「アメリカ合…」と右で切れる |
| 順位表 | 成績が「3-0-1（うち不戦…」と右で切れる |

E項・A〜I項でとってきた**「横スクロール+低優先列を隠す」という方針が限界**に達している。**残した列だけでも幅が足りない**ため、この延長線上では解決しない。

**方針: 640px以下ではテーブルレイアウトをやめ、1レコード=1カードの縦積みにする**(ユーザー決定 2026-07-22)。

#### O-1. 大会一覧(640px以下)

```text
────────────────────────────
MSG Radiant Origins Store Championship   ← 大会名(太字・折り返し可)
2026-07-19 ・ カナダ ・ Store Championship ← メタ行(折り返し可)
Magic Stronghold Games                    ← ホスト名(小さめ・グレー)
スタンダード ・ 20名                       ← 補足
────────────────────────────
```

- **`<thead>` は非表示**にし、各セルはラベル付きで縦に積む
- **スマホでフォーマット列(4列目)を隠す処理は廃止する**(カードなら収まるため、隠す必要がなくなる)
- カード全体をタップで大会詳細へ(既存の行クリック処理をそのまま活かす)
- **種別は略称化しない**(`Store Championship` のまま。収まらない場合は折り返す)。モックの「SC」は図示上の省略であって仕様ではない

#### O-2. 順位表(640px以下)

```text
────────────────────────────
🥇1  Fayrue                    📄 デッキ    ← 順位・プレイヤー名・デッキ導線
     勝ち点 7 ・ 2-0-1（うち不戦勝1）        ← 勝ち点と成績
────────────────────────────
🥈2  ik75                      📄 デッキ
     勝ち点 7 ・ 2-0-1
────────────────────────────
```

- **2行構成**。1行目 = 順位バッジ+プレイヤー名+デッキ導線(右寄せ)、2行目 = 勝ち点と成績
- **不戦勝の注記(`（うち不戦勝1）`)は省略しない**(J項の意図=順位の根拠を示すことが失われるため)
- GW%・MW%は引き続き非表示(現状維持)
- 上位3位の強調(`top1`/`top2`/`top3`)は維持する
- デッキ未提出の選手は導線を出さない(現状どおり)

#### O-3. 国名を通称に短縮する

C項で日本語国名にしたが、**正式名称が長すぎて列に収まらない**(`アメリカ合衆国`)。

- **日常的な通称に短縮する**(例: `アメリカ合衆国`→`アメリカ` / `大韓民国`→`韓国` / `グレートブリテン及び北アイルランド連合王国`→`イギリス`)
- **PC幅でも短縮名を使う**(列幅が安定し、一覧が読みやすくなるため)
- 収録実績のある31カ国すべてを見直す

#### 実装メモ

- **HTMLの変更は不要**。生成済みHTMLのセルには `.rank` / `.player-name` / `.score` / `.rec` 等のクラスが既に付いているため、**生成CSS内の `@media (max-width:640px)` だけで組み替えられる**
- カード内のラベル(「勝ち点」等)は `::before { content: ... }` か `data-label` 属性 + `content: attr(data-label)` で付与する。**いずれも生成CSS内で完結するため `style-src 'self'` に抵触しない**
- `<thead>` を隠す代わりにラベルを付けることで、スクリーンリーダーでの列の対応関係はむしろ改善する

## ユーザーによるローカル確認(2026-07-21追加)

実装・再ビルドの完了後、**pushの前に**ユーザー自身がローカルサーバーで実機確認を行う。開発担当は検証項目の結果をissueに報告したうえで、**ユーザーの承認が出るまでpushしない**。

```bash
npm run dev   # http://localhost:3000 (scripts/serve.mjs)
```

ユーザーが確認する観点(開発担当は報告コメントにこのチェックリストと、確認しやすい具体的URLを添える):

| # | 確認内容 | 確認先の例 |
| --- | --- | --- |
| U1 | 一覧から大会詳細への遷移・シーズン/カテゴリ/国/フォーマットの絞り込み | `/tournaments/` |
| U2 | 通常規模の大会でデッキがすぐ表示できる(**2026-07-22訂正: 旧「Ctrl+Fでカード名を全デッキ横断して探せる」は成立しない前提だったため削除**。A項参照) | `/tournaments/57690/` |
| U3 | **巨大大会でページが重くないか**(体感の最終判断。閾値100が妥当かの評価) | `/tournaments/51918/`(422デッキ) |
