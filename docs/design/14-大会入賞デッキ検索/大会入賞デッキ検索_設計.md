## 目的・差別化ポイント

大会(Regionals/Store Championship等)の入賞デッキを日本語訳付きで検索・閲覧できるようにする。**大会入賞結果を日本語訳付きで見られるサイトは現時点で他に存在しない**(2026-07-19時点調査)。カードDBの既存翻訳データをそのまま適用できるのは本サイトならではの強み。

参考事例: カードゲームショップCANTRIPのデッキ検索 <https://cantripmtg.com/pages/decks-grand-archive>(手動キュレーションによる自社ページ。出典表記なし)

## 調査結果(2026-07-19)

### 公式Omnidex APIは正式ドキュメント化されている(訂正)

issue #14登録時点では「未ドキュメントAPI」としていたが、**api-docs.gatcg.com に `/endpoints/omnidex` として正式収録されている**ことを確認(most updated 5ヶ月前)。以下がすべて公式ドキュメント記載のエンドポイント:

| エンドポイント | 内容 |
|---|---|
| `GET /omnidex/events/{event_id}` | 大会情報(名称/カテゴリ/フォーマット/日程/主催/形式) |
| `GET /omnidex/events/{event_id}/players` | 参加者(username/country/CP/世界rank/emblem) |
| `GET /omnidex/events/{event_id}/teams` | チーム戦の場合のチーム構成 |
| `GET /omnidex/events/{event_id}/judges` | ジャッジ一覧 |
| `GET /omnidex/events/{event_id}/standings` | 順位・勝敗・ゲーム勝率・タイブレーカー |
| `GET /omnidex/events/{event_id}/pairings` | ラウンドごとの対戦組み合わせ |
| `GET /omnidex/events/{event_id}/decklists` | **入賞者ではなく提出者全員のフルデッキ**(material/main/sideboard、英語カード名+枚数) |
| `GET /omnidex/events/{event_id}/statistics` | 大会の統計データ |

- **一覧・検索エンドポイントは存在しない**(idによる個別取得のみ)。イベントIDは連番で作成順に増加し、`api.gatcg.com/omnidex/events/{id}` を辿ってスキャンする以外に発見手段がない
- レート制限は公式ドキュメント上に明記なし。CORSは `*`(ブラウザから直接fetch可能)だが、無闇な高頻度アクセスは避け、キャッシュ・低並列を徹底する(既存 `tmp/ga-omnidex-mock/build.mjs` の並列数8を踏襲)
- 個人をIDで直接引くAPI(`/omnidex/players/{id}` 等)は存在しない。個人の戦績はイベント経由(`standings`)でのみ取得可能

### カテゴリ分布の実測(2026-07-19、1,200件スキャン)

| カテゴリ | 件数 | 割合 |
|---|---|---|
| regular(週次リーグ等) | 1,173 | 97.8% |
| store-championships | 24 | 2.0% |
| regionals | 3 | 0.25% |

- 欠番0件(スキャン範囲内でIDは概ね密に埋まっている)
- startAt範囲は2026-06-20〜2026-09-05(未来日程で先行登録される大会が一定数ある)
- 国別上位(1,200件中): US 371 / TW 158 / MY 106 / SG 77 / NZ 66 / **JP 65** / AU 62 / PH 54
- → **regular除外だけでデータ量が約1/50に減り、運用可能な規模になる**。nationals/invitational/worlds等の他カテゴリは今回のサンプル範囲では未検出だが、カテゴリはAPI側で自由文字列のため許可リスト方式(下記)で今後追加が容易

### 決定事項(2026-07-19ユーザー決定)

1. **対象大会**: **主要大会のみ・全世界**。カテゴリ許可リスト = `store-championships` / `regionals`(+今後判明次第 `nationals` / `invitational` / `worlds` 等を追加)。`regular` は除外
2. **実装方針**: **静的生成**(`scripts/build-card-pages.mjs` と同方式)。ビルド時にAPIをスキャンし、対象大会データをgit管理のJSONに保存→静的HTMLを事前生成。Cloudflare側の追加費用・サーバー処理なし。既存の「翻訳データはgit管理→pushで自動デプロイ」の運用にそのまま乗る

### 既存資産の再利用(カード名→slug解決)

デッキ構築ツールの「Omnidexインポート」機能([tools/deck-builder/app.js:1671-1714](../../../tools/deck-builder/app.js#L1671-L1714))が、まさに同じ課題(Omnidex提出フォーマットの英語カード名→slug解決)を解決済み:

- `normCardName()`: 記号・大小文字ゆれを吸収する正規化(`toLowerCase().replace(/[^a-z0-9]+/g,"")`)
- 完全一致のみ採用(曖昧一致は誤登録防止のため不採用)という方針も踏襲する

本機能はビルド時のバッチ処理のため、ライブAPI呼び出し(`resolveCardSlug`)ではなく**ローカルのカードスナップショット(`tmp/api-cache/cards-snapshot.json`)を使った同ロジックの一括解決**に置き換える(`scripts/build-card-pages.mjs` が同スナップショットを既に利用しているのと同じ考え方)。

### 読み取り専用デッキ表示との構造共通性

大会デッキは「編集不可」という点でデッキ構築ツールの共有画面(`#deck/<id>`)の表示ロジックに近い。データを `{card_slug, board(material/main/side), qty}` の形に正規化しておけば、統計タブの `computeDeckStats()` 等の集計ロジックを**そのまま入賞デッキにも適用できる可能性がある**(例: 「この大会の使用エレメント分布」)。ただし今回のMVPスコープでは表示のみとし、統計流用は将来検討(#3のコストカーブ検討と合わせて考える余地あり)。

## データモデル

### 保存先(git管理・data/tournaments/)

```text
data/tournaments/
├── events/
│   └── <event-id>.json      … 1大会分。取得済みの生データを正規化して保存
└── index.json                … 大会一覧(id/name/date/category/country/format/player数)。一覧ページ・スキャン再開の基準に使う
```

`events/<id>.json` の構造(案):

```json
{
  "id": 57690,
  "name": "CANTRIP RDOストアチャンピオンシップ",
  "category": "store-championships",
  "format": "standard",
  "startAt": "2026-07-12T03:30:00.000Z",
  "host": { "name": "CANTRIP", "country": "JP" },
  "playerCount": 22,
  "standings": [
    { "rank": 1, "player": 4571, "wins": 4, "losses": 0, "ties": 1, "gwPercent": 88.889, "mwPercent": 80 }
  ],
  "players": { "4571": "PlayerName" },
  "decklists": [
    {
      "player": 4571,
      "material": [{ "slug": "spirit-of-wind", "name": "Spirit of Wind", "qty": 1 }],
      "main": [{ "slug": "distilled-water", "name": "Distilled Water", "qty": 3 }],
      "sideboard": [{ "slug": null, "name": "未解決カード名", "qty": 1 }]
    }
  ]
}
```

- `slug: null` は名前解決に失敗したカード(表示は英語名のみ・リンクなし。ビルドログに警告出力)
- `index.json` は一覧ページと「次回スキャンの開始ID」判定の両方に使う(git管理されたファイル自体がチェックポイントを兼ねる。隠しキャッシュファイルを別途持たない)

### 静的ページ

```text
tournaments/
├── index.html            … 大会一覧(日付降順。カテゴリ/国/フォーマットは件数が少ないためクライアント側フィルタ)
└── <event-id>/
    └── index.html         … 大会詳細: 概要+順位表(standings)+全提出デッキ(デッキごとにmaterial/main/sideboard)
```

- 個別カードは `cards/<slug>/` への通常リンク。可能ならカードDBと同じ詳細モーダル(`shared/js/card-detail.js` + `card-i18n.js`)を読み込み、その場でポップアップ表示できるようにする(既存ページと同じUX)
- `sitemap.xml` への合流: `scripts/build-card-pages.mjs` のsitemap生成ステップが `tournaments/*/index.html` も走査して追加できるよう改修する(実行順序: 大会スキャン→カードページ生成、の順に実行する運用とする)

## 実装ステップ(実装担当セッション向け)

1. **スキャンスクリプト作成**: `scripts/build-tournament-pages.mjs`
   - `data/tournaments/index.json` から既知の最大event idを読み、そこから昇順(新規発見)+ 初回は指定の遡り基準(例: RDOシーズン開始 2026-04-10、`omni.gatcg.com/api/season` で取得可能)まで降順スキャン
   - 各idで `/omnidex/events/{id}` を取得。`category` が許可リスト(`store-championships`/`regionals`。定数で管理し追加しやすく)に一致する場合のみ `/players`・`/standings`・`/decklists` も取得
   - カード名→slug解決は `tmp/api-cache/cards-snapshot.json` を読み込み、`normCardName()` 相当のロジックで突き合わせ(deck-builder/app.jsの実装を移植・共通化を検討)
   - 取得結果を `data/tournaments/events/<id>.json` に保存、`index.json` を更新
   - 並列数は控えめ(6〜8程度)。進捗ログを出す(既存 `tmp/ga-omnidex-mock/build.mjs` を土台にする)
2. **静的ページ生成**: 同スクリプト内、または `build-card-pages.mjs` を拡張し、`data/tournaments/` から `tournaments/index.html`・`tournaments/<id>/index.html` を生成
3. **sitemap連携**: `build-card-pages.mjs` のsitemap生成ステップに `tournaments/` 配下のURLを追加
4. **CLAUDE.md/README更新**: 新しいnpmスクリプト(`npm run build:tournaments` 等)を追記

## 検証項目(実装時)

1. カテゴリフィルタが機能し、`regular` の大会が一切生成されないこと
2. 既知の実例(#57690 CANTRIP RDOストアチャンピオンシップ)で、22名分のデッキ・順位が正しく表示される
3. カード名→slug解決: 既存の全カード名で解決成功率を確認(未解決があればログで一覧化・原因確認)
4. 2回目以降の実行が「新規大会のみ」を追記し、既存の `data/tournaments/events/*.json` を書き換えないこと(冪等性)
5. `sitemap.xml` に `tournaments/` のURLが追加されていること
6. CSP・生成ページのアクセシビリティが既存の `cards/` ページと同水準であること
