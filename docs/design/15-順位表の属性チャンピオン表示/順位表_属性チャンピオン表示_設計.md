## 目的

大会の順位表は現在「順位 / プレイヤー / 勝ち点 / 成績 / GW% / MW% / デッキ」で、**そのデッキが何のデッキなのかがダイアログを開くまで分からない**。属性とチャンピオンを一覧できるようにして、大会の傾向把握とデッキ探しをやりやすくする。あわせてチャンピオン名で検索できるようにする。

## ユーザー要件(#14のローカル確認より・原文の要旨)

- 順位表の**プレイヤー名と勝ち点の間**に、属性アイコン → チャンピオン名の順で表示する
- 属性アイコンは**カード検索リストと同じ並び順**にする
- チャンピオン名は英語の前方部分＋日本語訳(例: `Diao Chan, Enchantress` → `Diao Chan（ダオ・チャン）`)
- チャンピオンが2種類入っている場合は**スラッシュで併記**
- **スピリットしか入っていない場合はそのスピリット名**を表示
- **チャンピオン名での検索**を追加する

## 画面モック

[モック_順位表.html](モック_順位表.html) — **実データ**(イベント#57690 CANTRIP RDOストアチャンピオンシップ・22名)で生成。ブラウザで直接開ける自己完結HTML。PC幅とスマホ幅(390px)の両方、および12属性の凡例を並べてある。

## 実測(2026-07-22・全420大会10,569デッキ)

### 属性は「チャンピオンの属性」を採用する

デッキ構築ツールの `deckDisplayElements()`([app.js:1827](../../../tools/deck-builder/app.js#L1827))は**デッキの全カード**のelementsを集める規則だが、これを順位表に使うと表示数が破綻する:

| 採用規則 | 属性の個数分布 |
|---|---|
| 全カードの和集合(deck-builderと同じ) | 1個=21% / 2個=46% / 3個=23% / **4〜10個=11%** |
| **チャンピオン(スピリット含む)のみ** | **1個=38% / 2個=62%**(3個=1件・0個=1件) |

デッキ構築ツールはデッキ全体を大きく表示する画面なので前者で問題ないが、**順位表の1行に10個の玉は載らない**。GAではデッキの識別子はチャンピオンの属性なので、**material内の`types`に`CHAMPION`を含むカード(スピリット含む)のelementsのみ**を採用する。

- 並び順は既存規則を踏襲: **基本属性(`NORM`/`FIRE`/`WATER`/`WIND`の順)→上級属性はアルファベット順**。`EXALTED`は独自の玉を持たないため除外。`NORM`は他が無いときのみ表示([card-search.js:41](../../../shared/js/card-search.js#L41)・[app.js:500](../../../tools/deck-builder/app.js#L500)と同じ)
- 出現する属性は**12種**: FIRE 3,767 / WIND 3,405 / WATER 3,370 / LUXEM 1,688 / UMBRA 1,508 / CRUX 1,264 / TERA 788 / EXIA 494 / ASTRA 344 / ARCANE 251 / NEOS 188 / NORM 61
- 属性0個のデッキが1件だけある → **フォールバック表示(`—`)**が必要

### チャンピオン名の抽出

| ケース | 件数 | 扱い |
|---|---|---|
| 通常(非スピリットのチャンピオンあり) | 10,449件 | そのチャンピオン名 |
| チャンピオン2種以上 | 410件 | **スラッシュ併記**(例: `Alice（アリス） / Merlin（マーリン）`) |
| スピリットのみ | 119件 | **スピリット名**(例: `Spirit of Wind（風の精霊）`) |
| どちらも無し | 1件 | **`—`** |

- 判定: `types` に `CHAMPION` を含む → チャンピオン。うち `subtypes` に `SPIRIT` を含む → スピリット
- 表示名 = **英語名の最初のカンマまで** ＋ `（日本語名の最初の読点まで）`。日本語訳が英語と同じなら括弧を出さない
- **レベル違いは前方部分で自動的に集約される**(`Alice, Distorted Queen` / `Alice, Golden Queen` → ともに `Alice（アリス）`)。重複排除はこの表示名で行う
- **(2026-07-22追加)表示は最大3件まで。超える場合は「先頭3件 ＋ `他N種`」**とし、`title` 属性には全件を入れる。実測の分布は **0件=1 / 1件=10,157(96%) / 2件=409 / 3件=1 / 12件=1** で、**3件までを全表示にすれば実質1デッキ(大会#53836・全材料がスピリット12種)だけが省略対象**になる。この1件は省略しないとスマホ幅で1行が約400pxを占める(表示長413文字)

## 属性アイコンの実現方式(採用: ビルド済みWebPをdata URIでCSSに埋め込む)

既存の属性玉はカード画像から円形に切り出す実装だが、**canvas描画のデッキサムネイル専用**([app.js:1856](../../../tools/deck-builder/app.js#L1856))で、順位表(最大422行)にはそのまま使えない。4案を比較した:

| 案 | 見た目 | 転送量 | 判定 |
|---|---|---|---|
| 1. CSSでカード画像から玉を切り出す | デッキ構築ツールと同一 | **属性ごとにカード画像1枚=約100KB。12属性で最大1.2MB** | ページ軽量化(#14で最大402KBまで削減)を台無しにする |
| 2. 文字バッジ(火/水/風…) | 独自 | ゼロ | 上級属性が「ラクサム」「アンブラ」等4文字。2属性で8文字超になり幅を食う。1文字略記は**アが3つ衝突**(アンブラ/アストラ/アーケイン)して不可 |
| 3. 色付きSVG | 独自・サイト統一感なし | ほぼゼロ | 12色を色だけで判別するのは無理 |
| **4. 玉を切り出した画像をビルド済みでdata URIに** | **1と同一** | **12属性合計 約10KB** | **採用** |

### 案4の詳細

- 切り出し座標は既存実装と同じ(**500×700基準で中心449,47・半径22**)。元画像は各属性の**単属性カード**を使う(複属性カードは玉の周りにEXALTEDの金装飾等が写り込むため。[app.js:1841](../../../tools/deck-builder/app.js#L1841) の `orbSourceCard()` と同じ考え方)
- **32px・WebP・quality 80**で書き出し、base64のdata URIとして `tournaments.css` に**12クラス分だけ**埋め込む(`.orb-fire` 等)。実測 **合計10KB**
- 各行は `<i class="orb orb-fire" role="img" aria-label="火"></i>` を並べるだけ。**外部リクエストは一切発生しない**
- **生成物はリポジトリにコミットする**(定数ファイル)。生成には `python3-pil` が必要だがこれはaptパッケージで**コンテナ再構築で消える**(CLAUDE.md参照)ため、**ビルド(`npm run build:tournaments`)からは切り離す**。新しい属性が追加されたときだけ手動で再生成する
- 切り出し元カードの実測(12属性すべて単属性カードあり): FIRE=`absolving-flames` / WIND=`adept-swordmaster` / WATER=`acquiescing-rejection` / LUXEM=`advent-of-the-shenju` / UMBRA=`abnegation` / CRUX=`beacon-knight` / TERA=`acerbica` / EXIA=`annihilation` / ASTRA=`aethercloak-sentinel` / ARCANE=`advent-of-the-stormcaller` / NEOS=`aegis-of-dawn` / NORM=`academy-attendant`

### 生成スクリプト

設計時のプロトタイプ(PIL使用・12属性を32px WebPのbase64にしてJSON出力)は動作確認済み。実装時は `scripts/gen-element-orbs.mjs` 等として整備し、出力を `scripts/lib/element-orbs.json`(または `.mjs`)にコミットする。**日次cronのビルドはこのコミット済みファイルを読むだけ**にすること。

## 表示仕様

### PC幅

`プレイヤー` と `勝ち点` の間に**「デッキ」列**を追加する。

```text
順位 | プレイヤー | デッキ                          | 勝ち点 | 成績  | GW% | MW% |
🥇1  | yokomi     | ●● Arisanna（アリサナ）          |   13   | 4-0-1 | 89% | 80% | 📄
```

- 属性玉 → チャンピオン名の順。玉は18px・円形・細い暗色の縁
- チャンピオン名が長い場合は**省略記号で詰める**(`text-overflow: ellipsis`)。`title` 属性に全文を入れる
- 玉には `role="img"` と `aria-label="火"` 等を付ける(スクリーンリーダー対応)

### スマホ幅(640px以下)

#14のO-2で2行カードにしたので、**デッキ情報を独立した行として挟み、3行構成**にする。

```text
────────────────────────────
🥇1  yokomi                    📄 デッキ
●● Arisanna（アリサナ）
     勝ち点 13 ・ 4-0-1
────────────────────────────
```

- 属性・チャンピオンは**省略しない**(この機能の主目的のため)。長い場合は折り返す
- 既存の2行(順位・名前・導線 / 勝ち点・成績)の構成は維持する

## チャンピオン名での検索

**2箇所に入れる**(ユーザー要望の「検索に追加」と、#14で保留した「大会内のプレイヤー検索」を同時に満たす)。

### 1. 大会一覧のフリーワード検索(既存 `data-search` に追加)

- 各大会行の `data-search` に、その大会に出現した**チャンピオン名(重複排除)**を追加する
- 「`Arisanna` を使った人がいる大会」を一覧から探せるようになる
- 既存の大会ID・大会名・ホスト名・プレイヤー名検索([#14](https://github.com/dozyouneko/ga-card-tools-jp/issues/14) H項)と同じ入力欄で動く

### 2. 大会詳細の順位表に絞り込み入力を新設

- 順位表の上に**1つの入力欄**を置き、**プレイヤー名・チャンピオン名・属性**で行を絞り込む(クライアント側)
- **#14で保留した「大会内のプレイヤー検索」はこれで解決する**([#14の確認事項①](https://github.com/dozyouneko/ga-card-tools-jp/issues/14))
- ⚠️ **100行ごとの折りたたみ(#14 B項)との相互作用**: 絞り込み時は**折りたたみを無視して該当行だけを平坦に表示**する。折りたたんだブロックの中に隠れると検索の意味がなくなるため
- 絞り込み結果が0件のときは「該当する選手がいません」を表示する

## 実装ステップ

1. **属性玉の生成スクリプト**: `scripts/gen-element-orbs.mjs`(PIL相当の処理。Node実装が難しければ設計時プロトタイプのPythonを整備してもよい)。出力をコミットする
2. **データ側**: `collectEvent()` で各デッキの**チャンピオン表示名と属性リストを導出して `events/<id>.json` に保存**する。ページ生成のたびにカードスナップショットを引き直さずに済み、一覧の `data-search` 生成も楽になる。⚠️ **`--refresh` による再取得ではなくページ再生成時に導出でも可**。どちらにするかは実装判断でよい(保存する場合はスキーマ追加を報告すること)
3. **順位表**: 「デッキ」列を追加(PC)。スマホ幅は3行カードに拡張
4. **CSS**: `.orb` と12個の `.orb-*` クラスを `tournaments.css` に追加
5. **検索**: 一覧の `data-search` にチャンピオン名を追加 / 大会詳細に順位表の絞り込み入力を新設
6. **再ビルド**: データにチャンピオン情報を保存する場合は再取得が必要。導出のみなら `--no-scan` で足りる

## 検証項目

1. 属性玉が**チャンピオンの属性**に基づいており、1デッキあたり**1〜2個**に収まっていること(実測で3個は1件のみ)。並び順が基本属性→上級属性のアルファベット順であること
2. 属性0個のデッキ(1件)とチャンピオン0件のデッキ(1件)で `—` にフォールバックし、レイアウトが崩れないこと
3. チャンピオン表示が仕様どおりであること: 通常=`Arisanna（アリサナ）` / 2種=スラッシュ併記(409件) / スピリットのみ=スピリット名(119件) / レベル違いが同一表示に集約されること。**(2026-07-22追加)** 4件以上のデッキ(大会#53836の12件)が「先頭3件 ＋ `他N種`」に省略され、`title` に全件が入っていること。3件のデッキ(大会#50946)は省略されないこと
4. `tournaments.css` に埋め込んだ属性玉が**12種で合計15KB以内**に収まっていること。**外部への画像リクエストが発生しないこと**(DevToolsのNetworkで確認)
5. 属性玉に `aria-label` があり、スクリーンリーダーで属性名が読めること
6. PC幅で「デッキ」列が表示され、長いチャンピオン名が省略記号で詰められ `title` に全文があること
7. スマホ幅(390px)で3行カードになり、**属性・チャンピオンが省略されずに表示**され、横スクロールが発生しないこと
8. 大会一覧のフリーワード検索で**チャンピオン名(例: `Arisanna`)がヒット**すること。既存の大会ID・大会名・プレイヤー名検索が壊れていないこと
9. 大会詳細の絞り込みでプレイヤー名・チャンピオン名・属性が引けること。**絞り込み中は100行ごとの折りたたみを無視して該当行が平坦に出る**こと。0件時のメッセージが出ること
10. ページ生成が**日次cronのビルドで完結する**こと(PILやネットワークに依存していないこと)
11. **(2026-07-22訂正)** 大会詳細ページのサイズ増加が許容範囲であること。**旧記述「#14時点で中央値12.8KB・最大402KB」は誤り**(402KBは一覧ページ`tournaments/index.html`の値だった)。正しい#14時点の実測は **中央値12.5KB・最大148.4KB**(`origin/main` の `tournaments/<id>/index.html` 420件)。判定は**転送実体であるgzip後の値**で行う(生の値ではなくCloudflareが配信するサイズで評価する)
