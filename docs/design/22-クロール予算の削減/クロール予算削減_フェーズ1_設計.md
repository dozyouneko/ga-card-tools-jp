### 遅延化が容易な理由 — 依存が既に切れている

`shared/pdf/card-sheet.js` は **jsPDFをトップレベルで参照していない**。
コンストラクタを**引数で受け取る**設計になっている。

```js
// shared/pdf/card-sheet.js
function buildPdfPaged(images, opt, jsPDF) {
  if (!jsPDF) throw new Error("jsPDF constructor is required");
```

つまり **`card-sheet.js` は jsPDF なしで読み込める**。切り離しに必要なのは
「呼ぶ直前に jsPDF を用意する」ことだけ。

### ⚠️ `card-sheet.js`(9 KB)は遅延化しない

`CardSheet.pageCountFor()` が**ページ読み込み時に呼ばれ得る**ため。

```js
// app.js:312 updatePrintBar() — init() の最後(app.js:601)から呼ばれる
function updatePrintBar() {
  const n = totalCards();
  if (n === 0) { bar.hidden = true; return; }   // ← 印刷リストが空なら CardSheet に触れない
  const pages = CardSheet.pageCountFor(n);      // ← 項目があるとここで必要になる
```

Googlebotは印刷リストが常に空なので早期returnするが、**項目を持つ実ユーザーは読み込み時に必要**。
9 KB(全体の0.55%)しかないため、**eagerのまま据え置く**のが安全かつ費用対効果が高い。

## 設計

### A. トップページ: jsPDF を遅延読み込みにする

**A-1. [index.html:127](../../../index.html#L127) の `<script>` を削除する。**
`card-sheet.js`(128行目)は**残す**。

**A-2. [app.js](../../../app.js) にローダーを追加する。**

```js
// jsPDF(356KB)はPDF生成のときだけ要る。トップページの初期読み込みから外し、
// 「PDF生成」を押した時点で取得する（クロール予算とスマホの初期表示を軽くするため）
let jspdfPromise = null;
function loadJsPdf() {
  if (jspdfPromise) return jspdfPromise;
  jspdfPromise = new Promise((resolve, reject) => {
    const s = document.createElement("script");
    s.src = "shared/vendor/jspdf.umd.min.js";
    s.onload = () => resolve(window.jspdf);
    s.onerror = () => { jspdfPromise = null; reject(new Error("PDFエンジンの読み込みに失敗しました")); };
    document.head.appendChild(s);
  });
  return jspdfPromise;
}
```

- **失敗時に `jspdfPromise` を null に戻す**こと。戻さないと一度失敗したら再試行できなくなる
- `src` は**既存の書き方に合わせた相対パス**でよい(トップページは `/` 配信のため `/shared/vendor/...` に解決される)

**A-3. [app.js:363](../../../app.js#L363) の取得方法を差し替える。**

```js
// 変更前: const { jsPDF } = window.jspdf;
status.textContent = "PDFエンジンを読み込み中…";
const { jsPDF } = await loadJsPdf();
```

`generateProxyPdf()` は**既に `async`** なので `await` をそのまま置ける。
例外は既存の `try/catch`(`status.textContent = "❌ エラー: " + err.message`)が拾う。

### B. `/tools/print/`: 同様に遅延化する ⚠️ こちらは構造変更が要る

[tools/print/app.js:8](../../../tools/print/app.js#L8) は **トップレベルで分解代入**しており、
`<script>` を外すと**その場で `window.jspdf` が undefined になり全体が壊れる**。

```js
const { jsPDF } = window.jspdf;   // ← モジュール先頭。ここを消す必要がある
```

さらに生成ハンドラが**同期**である点も異なる。

```js
generateBtn.addEventListener("click", () => {
  setStatus("生成中…");
  setTimeout(() => {
    try { const pdf = CardSheet.buildPdf(images(), getOptions(), jsPDF); ... }
```

**対処**:

1. `tools/print/index.html` から `<script src="../../shared/vendor/jspdf.umd.min.js">` を削除
2. トップレベルの `const { jsPDF } = window.jspdf;` を**削除**し、A-2と同じローダーを置く
   (`src` は `../../shared/vendor/jspdf.umd.min.js`)
3. `generateBtn` のハンドラを **`async`** にし、`setTimeout` の中ではなく
   **クリック直後に `await loadJsPdf()`** してから既存の生成処理に入る

⚠️ **`setTimeout(…, 30)` は「生成中…」を描画させるための意図的なウェイト**。
`await` を入れると同じ効果が得られるため `setTimeout` は残しても外してもよいが、
**残す場合は `await` の完了後に呼ぶ**こと(順序を逆にすると `jsPDF` が未定義のまま入る)。

### C. `_headers` にベンダーのキャッシュ指定を追加する

```
# jsPDFは遅延読み込み(#22)。90日で1回しか変わっていないベンダーライブラリのため長期キャッシュする。
# ⚠️ 差し替えるときは必ずファイル名を変えること(このヘッダのままだと更新が届かない)
/shared/vendor/*
  Cache-Control: public, max-age=31536000, immutable
```

**⚠️ この項目の目的はSEOではなくUXである。** jsPDFを遅延化した時点で
**Googlebotはこのファイルを取得しなくなる**ため、クロール予算への効果はゼロ。
狙いは「PDFを2回目以降に生成する実ユーザーが355 KBを再取得しない」こと。

**`data/tl/*.js` にはキャッシュ指定を入れない(フェーズ1では)。** 理由:

- 90日で**110回**変更されており、ファイル名にハッシュが無いため**古い翻訳が配信され続ける実害**がある
- Googlebotのレンダラが `Cache-Control` をどこまで尊重するかは**確認できていない**
  (Googleはレンダリング用リソースを独自基準でキャッシュすると説明している)。効果が不確実
- **フェーズ2でこの38本の構成自体を見直す**ため、いま入れたTTL設計は作り直しになる

## 期待効果 — ⚠️ フェーズ1だけでは解決しない

| ページ | 現状 | フェーズ1後 | 削減 |
|---|---|---|---|
| `/`(トップ) | 46本 / **1,636 KB** | 45本 / **1,281 KB** | **−355 KB(−21.7%)** |
| `/tools/print/` | 3本 / **368 KB** | 2本 / **13 KB** | **−355 KB(−96.5%)** |

**残る最大の要因は `data/tl/*.js` の 1,162 KB** で、フェーズ1後のトップページの
**90.7%** を占める。Googleが10日間でダウンロードした総量は1.8 MBであり、
**フェーズ1後もトップ1枚のレンダリングで1.28 MBを消費する**。

**インデックスされない状態の根本解決にはフェーズ2が必要**である。
本フェーズは「安全に取れる2割を先に取る」ことと、`/tools/print/` の実質解消が目的。

### 実ユーザーへの効果も実測した(2026-07-23)

`<main id="grid">` は**空のまま配信され、カード一覧はJSが描画する**。
scriptは `</body>` 直前にあるが **`defer` が無い**ため、**46本すべてが読み終わるまでカードは1枚も出ない**。

Playwright + CDPで**3G相当(1.6 Mbps・遅延150ms)・390px**に絞って計測:

| | DOMContentLoaded | 転送量 |
|---|---|---|
| 現状 | **15,651 ms** | 1,684 KB |
| jsPDF遅延化後 | **13,900 ms** | 1,328 KB |
| 差 | **−1,751 ms** | −356 KB |

⚠️ **改善しても13.9秒かかる。** スマホ回線で「カードが出るまで14秒」は実用的とは言えず、
**フェーズ2(`data/tl/*.js` 1,162 KB)の必要性を裏づける数字**である。
`defer` の付与も併せてフェーズ2で検討する。

## 実装ステップ

1. **`index.html`**: 127行目の jsPDF `<script>` を削除(`card-sheet.js` は残す)
2. **`app.js`**: `loadJsPdf()` を追加し、`generateProxyPdf()` の `window.jspdf` 参照を `await loadJsPdf()` に置換
3. **`tools/print/index.html`**: jsPDF `<script>` を削除
4. **`tools/print/app.js`**: トップレベルの分解代入を削除、ローダーを追加、生成ハンドラを `async` 化
5. **`_headers`**: `/shared/vendor/*` のキャッシュ指定を追加

⚠️ **生成物の再ビルドは不要**。`index.html` はテンプレートではなく実体で、
カード個別ページ・大会ページは jsPDF を読み込んでいない(実測確認済み)。

## 未決定・確認事項

- **フェーズ2(`data/tl/*.js` 1,162 KB)の設計は本issueに残る。** 制約として、
  `GA_I18N.cards` は日本語ローカル検索([card-search.js](../../../shared/js/card-search.js) の `localJpSlugs()`)が
  **全カードを走査する**ため、単純な遅延化・分割は日本語検索を壊す。
  「どの機能がいつ全件を必要とするか」の整理が先。**フェーズ1の本番反映後に着手する**
- **P5の観察結果次第でフェーズ2の優先度を再判断する。** クロール統計の改善が見られなければ、
  `data/tl` の削減効果も限定的である可能性があり、[#6](https://github.com/dozyouneko/ga-card-tools-jp/issues/6)(Next.js移行によるSEO強化)を
  含めた別のアプローチを検討する必要がある
- `/tools/print/` は**PDF生成が主目的のページ**のため、遅延化すると初回生成時に必ず待ちが入る。
  Z3で体感が悪ければ「画像を1枚追加した時点で先読みする」案に切り替える(実装は `loadJsPdf()` を呼ぶだけ)
- [#23](https://github.com/dozyouneko/ga-card-tools-jp/issues/23)(soft 404)と**同じリリースにまとめると本番確認が1回で済む**。
  どちらも `curl -sI` での実物確認が必要なため
