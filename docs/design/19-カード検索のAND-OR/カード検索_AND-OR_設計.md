# issue#19: カード検索にAND/OR条件を追加する 設計書

GitHub issue: [#19](https://github.com/dozyouneko/ga-card-tools-jp/issues/19)

| 日付 | 版 | 内容 |
|---|---|---|
| 2026-07-22 | 初版+モック | ユーザー要望(依頼元の原文「norm or wind みたいな感じ」)を設計化。対象はエレメント/クラス/タイプ/サブタイプの4項目、AND・OR両対応(ユーザー決定)。**公式APIは複数値をORとしてのみ処理する**ことを実測で確認したため、**ANDはクライアント側で実装する**方針を確定。カード2,240枚の実データで各項目の複数持ち分布を測り、ANDの有用性が項目ごとに大きく違うことを明記。**サブタイプが146種あるため既定は上位20種のみ表示**する方針を追加(上位20種で全カードの100%をカバー)。PC幅/スマホ幅を並べたモックを作成。**既存の外側トグル(スマホ幅で絞り込み全体を畳む挙動)は維持**し、各グループは既定で閉じる方針を追加(4グループを開くとチップが50個以上並びカード一覧が押し出されるため) |

## 目的・要望

カード検索の絞り込みで、**同一項目に複数の値を指定**できるようにする。依頼元の原文は「**norm or wind みたいな感じ**」。

**ユーザー決定(2026-07-22)**:

- 対象項目は **エレメント / クラス / タイプ / サブタイプ** の4つ。エレメント以外も**必須**
- 4項目とも **AND・OR の両方**に対応する
- **AND/ORを切り替えるトグル**を各リストの近くに置き、複数選択した値をトグルの指定方法で検索する
- エレメントのANDは「基本4属性とEXALTEDの組み合わせ」が存在するため必要。**ANDで該当し得ない組み合わせが指定されたらメッセージを表示する**

## 画面モック

[モック_絞り込みUI.html](モック_絞り込みUI.html) — ブラウザで直接開ける自己完結HTML。**PC幅とスマホ幅(390px)を並べて**確認できる。エレメントの玉は[#15](https://github.com/dozyouneko/ga-card-tools-jp/issues/15)で作成した実物のアイコン(32px WebP・data URI)を流用している。下段には**エレメントANDで成立しない組み合わせを選んだときの警告表示**も掲載。

※見た目の確認用で操作はできない。

## 実測(2026-07-22・カードスナップショット2,240枚)

### 項目ごとの「複数持ち」分布 — ANDの有用性は項目で大きく違う

| 項目 | 1個 | 2個 | 3個以上 | ANDの有用性 |
|---|---|---|---|---|
| elements | 2,167 | 70 | 0 | **限定的**(下記) |
| classes | 2,091 | 149 | 0 | あり |
| types | 1,674 | 560 | 6 | あり |
| subtypes | 32 | 1,571 | 637 | **非常に高い**(99%が複数持ち) |

サブタイプは `Abnegation[CLERIC+SPECTER+SPELL]` のように**ほぼ全カードが複数持ち**で、AND検索の価値が最も高い。タイプも `Aegis of Dawn[REGALIA+ITEM]` のように複数持ちが566枚ある。

### エレメントのANDは4通りしか成立しない

**複属性カード70枚はすべて `EXALTED + 基本属性` の形**で、EXALTEDを含まない組み合わせ(`FIRE+WATER` 等)は**1枚も存在しない**。

| 組み合わせ | 枚数 |
|---|---|
| EXALTED+NORM | 31 |
| EXALTED+WIND | 15 |
| EXALTED+FIRE | 12 |
| EXALTED+WATER | 12 |

したがって**エレメントのANDが0件にならないのは上記4通りのみ**。「FIRE AND WIND」「EXALTED AND CRUX」「3つ以上のAND」は**構造的に必ず0件**になる。ユーザー要望の「メッセージを表示する」はこの状況への対処と解釈する(2026-07-22 ユーザー確認済み)。

### 公式APIは複数値を**ORとしてのみ**処理する(AND非対応)

同名パラメータを繰り返すとORになることを実測で確認した。

```text
element=FIRE 297件 / element=WATER 345件 / element=FIRE&element=WATER 642件  → OR
subtype=CLERIC 581件 / subtype=SPELL 550件 / subtype=CLERIC&subtype=SPELL 896件 → OR
（ローカル実測: CLERIC AND SPELL は 235件。APIからは取得できない）
```

**ORはAPIに任せられるが、ANDはクライアント側で実装するしかない。** これが本設計の最大の制約。

## 実装方針

### OR: APIに委譲(現行の延長)

`buildQuery()` で選択値の数だけ `p.append("element", v)` を繰り返す。**エキスパンション絞り込み(`setPrefixes()`)が既に同じ方式**なので前例がある。サーバー側でページングされるため件数・ページ送りは正確なまま。

### AND: 1つの値でAPIを引き、残りをクライアント側で絞る

APIがANDに非対応のため、以下の方式を採る。

1. 選択値のうち**1つだけ**をAPIに送る(結果はANDの結果を必ず包含する上位集合)
2. 返ってきたページに対し、**残りの選択値をすべて含むか**をクライアント側で判定して間引く
3. 間引いた結果を表示する

- **既存の日本語テキストモード(`localJpSlugs()` + `matchesActiveFilters()`)と同じ考え方**で、クライアント側の後段フィルタという構造は既にこのコードにある
- APIリクエストは**1ページにつき1回**で済む(スラッグ単位で引き直す日本語モードの方式より公式APIへの負荷が軽い。CLAUDE.mdの「低並列を徹底する」方針に沿う)
- ⚠️ **ページごとの表示件数が不揃いになる**(50件取得して13件しか残らない等)。本サイトは「もっと見る」方式なので致命的ではないが、**総件数は正確に出せない**

### 件数表示の扱い

AND時は総件数が確定できないため、**「表示中 N件」のように実際に描画した件数を出す**か、上位集合の件数であることが分かる表記にする。

**(2026-07-22確定)** 既存の `N 件を表示 / 全 N 件` の表記を維持し、総件数を出せない場合のみ注記を添える。モックに書いた `検索結果：N件` は[#18](https://github.com/dozyouneko/ga-card-tools-jp/issues/18)(大会デッキ検索)の表記であり、**カードDBは対象外**なので寄せない。あわせて**日本語モードで従来出ていた不正確な「全 N 件」を出さないように直す**(既存の `total = jpSlugs.length` の制約への対処)。

**現行コードにも同じ制約が既にある**([card-search.js](../../../shared/js/card-search.js) の日本語モード: `total = jpSlugs.length; // 追加の絞り込みは客側適用のため、総数には反映されない`)。本対応で表記を統一するとよい。

### 検討したが採らない案: カードメタデータ索引を配信する

slug→{elements, classes, types, subtypes} の索引を生成して配信すれば、AND判定を完全にクライアント側で行え、**総件数も正確に出せる**。サイズは実測で **173KB(gzip 30KB)** と現実的だった。

しかし**ビルド時点のスナップショットに固定されるため、新カードが出てから再ビルドするまで索引が古くなる**。APIは常に最新を返すのに索引だけ遅れる状態は分かりにくい。**まずは索引なしで実装し、総件数の正確さが問題になった時点で再検討する**。

## UI設計

### 開閉の構造 — 既存の外側トグルは維持し、グループは既定で閉じる

現行のカード検索は**スマホ幅(≤640px)で絞り込み全体が「絞り込み・並び替え ▾」に畳まれる**([style.css:317](../../../style.css#L317) の `.filter-row { display:none }` / `.controls.filters-open .filter-row`)。PC幅では `.filter-toggle` が `display:none` で常時表示。**この挙動は変更しない。**

そのうえで、**各項目のグループ(エレメント/クラス/タイプ/サブタイプ)は既定で閉じる**。

- 4グループを開いたままにすると、チップが合計50個以上並んでパネルが極端に長くなり、**カード一覧が画面外に押し出される**
- 閉じていても**選択中の件数をバッジで表示**するため状態は分かる
- 必要なグループだけ開く運用にする。PC幅・スマホ幅とも同じ挙動でよい

結果として**スマホは2段構え**になる:

| 段 | 状態 |
|---|---|
| ① 既定 | 「絞り込み・並び替え ▾」だけが見える(現行どおり) |
| ② 開いた直後 | 4グループの見出し+フォーマット/エキスパンション/並び替え/リセットが並ぶ。**チップは出ない** |
| ③ グループを開く | そのグループのチップだけが展開される |

### (2026-07-22追加)PC幅ではグループを横に並べる

初回実装では**4グループがそれぞれ全幅1行を占めて縦に積まれ**、検索入力2行と合わせて**6行**が絞り込みに使われてカード一覧が大きく下へ押し出された(ユーザー指摘)。グループ見出しの右側に1,000px以上の空白ができるのも無駄。

**PC幅(641px以上)ではグループをグリッドで横に並べる。**

```css
@media (min-width: 641px) {
  #filter-row { display: grid; grid-template-columns: repeat(auto-fit, minmax(240px, 1fr)); gap: 8px; align-items: start; }
  .fgroup + .fgroup { margin-top: 0; }
  .fgroup[open] { grid-column: 1 / -1; }   /* 開いたグループだけ全幅に広がる */
  .filter-rest { grid-column: 1 / -1; }
}
```

- **閉じているときは4グループが1行に収まる**(絞り込み全体が2行になり、カード一覧が上がる)
- **開いたグループだけ `grid-column: 1 / -1` で全幅に広がる**ためチップが窮屈にならない。残りのグループは次の行にタイル状に並ぶ
- `auto-fit` + `minmax(240px, 1fr)` により**幅に応じて列数が変わる**。実測: 700px→2列 / 900px→3列 / 1180px→4列。**いずれも横スクロールは発生しない**
- スマホ幅(≤640px)は現行どおり縦積み(この `@media` の対象外)

### 複数選択の形式

現行の `<select>`(単一選択)を複数選択できる形に変える。**`<select multiple>` はスマホで扱いにくいため採用しない。**

- **チェックボックス群**を基本とする。項目名をタップで開閉するトグル(`<details>`)に入れ、既定は閉じる
- **エレメントは13種と少なく、[#15](https://github.com/dozyouneko/ga-card-tools-jp/issues/15)で属性玉のアイコン(32px WebP・data URI)を作成済み**なので、**玉アイコン付きのトグルボタン群**にすると視認性が上がりカードDBらしくなる
- ⚠️ **`scripts/lib/element-orbs.json` はNode側のファイルでブラウザから読めない**。CSPが `style-src 'self'` でインラインstyleも使えないため、**JSONからCSSファイルを生成してコミットし `<link>` で読み込む**(`scripts/gen-element-orbs-css.mjs` → `shared/css/element-orbs.css`)。sharp・ネットワークが要る `gen-element-orbs.mjs` とは別物で、**JSONを読むだけでオフライン実行できる**
- ⚠️ **EXALTEDには玉がない**(`element-orbs.json` は12種でEXALTEDを含まない)。これはゲーム上もEXALTEDが独自の玉を持たない(金装飾表現)ためで、[deck-builder/app.js](../../../tools/deck-builder/app.js) の `deckDisplayElements()` も同じ理由でEXALTEDを除外している。**EXALTEDのチップはテキストのみとする**
- 選択中の件数をバッジ等で示す(閉じていても状態が分かるように)

### サブタイプは146種あるため既定では上位20種のみ出す

サブタイプは**146種**あり、全部をチップで並べると一覧性が失われる(クラス9種・タイプ15種・エレメント13種は全件並べて問題ない)。

実測すると**出現頻度の偏りが大きく、上位20種で全2,240枚の100%をカバー**する(上位10種でも98%)。一方で**1枚しか該当しないサブタイプが29種**ある。

| 上位N種 | カバーされるカード |
|---|---|
| 10種 | 2,202 / 2,240(98%) |
| **20種** | **2,239 / 2,240(100%)** |

- **既定は上位20種のみ**をチップで表示する
- 「**＋ すべて表示（146種）**」で残りを開く
- 表示順は**出現頻度の降順**とする(実装時にスナップショットから算出)
- **(2026-07-22訂正)** 初版に書いた上位20種の列挙は**末尾5種が誤り**だった(カバー率2,205止まり)。正しい頻度降順の上位20種は次のとおりで、カバー率2,239(=表の値と一致):
  `CLERIC / SPELL / HUMAN / MAGE / TAMER / WARRIOR / GUARDIAN / SKILL / RANGER / ASSASSIN / REACTION / SWORD / ANIMAL / ACCESSORY / AUTOMATON / ARTIFACT / SPECTER / BEAST / CHESSMAN / SPIRIT`
- **(2026-07-22追記)** 絞り込み候補は**カードスナップショットではなく `I18N.meta.subtypes` から生成する**(既存の `fillSelect()` と同じ挙動)。そのため UI 上は**147種**になる(差分は `SHENJU`。訳語はあるがスナップショット上どのカードにも付いていない)。ボタン表記は「＋ すべて表示（147種）」

### AND/ORトグル

- **項目ごとに1つずつ**置く(エレメントはOR・サブタイプはAND、のような混在ができる)
- 2値のセグメント切り替え(`AND` / `OR`)。**既定はOR**(「norm or wind」という要望の主目的であり、ANDより直感的なため)
- **(2026-07-22確定)** 値を1つ以下しか選んでいないときも**トグルは常時有効**にする。無効化すると「先にANDを選んでから値を選ぶ」操作ができず不便なため(1値ならAND/ORで結果は変わらないので実害はない)

### エレメントAND時のメッセージ

構造的に0件になる組み合わせが指定されたら、検索を実行する前にメッセージを出す。

> この組み合わせに該当するカードはありません。複数エレメントを持つカードは「EXALTED＋基本属性(NORM/FIRE/WATER/WIND)」の組み合わせのみです。

## 実装ステップ

1. **UI**: `#f-element` / `#f-class` / `#f-type` / `#f-subtype` を複数選択(チェックボックス群・エレメントは玉アイコン付きトグル)に変更。各項目にAND/ORトグルを追加
2. **クエリ生成**: `buildQuery()` を複数値対応にする。**OR時は選択数だけ `p.append()`、AND時は1つだけ送る**
3. **クライアント側フィルタ**: `matchesActiveFilters()` を単一値比較から**複数値+AND/OR判定**に変更する。⚠️ **日本語テキストモードもこの関数を通る**ため、両モードで同じ絞り込み結果になることを必ず確認する
4. **AND時の後段フィルタ**: APIレスポンスの各ページに対して残りの条件を適用する処理を追加
5. **件数表示**: AND時は総件数が出せないため表記を調整(日本語モードの既存の制約と統一)
6. **エレメントANDのメッセージ**: 成立し得ない組み合わせの判定と表示
7. **URLクエリ**: 絞り込み状態の共有・復元があれば複数値+AND/ORも含める

## 検証項目

1. **OR**: エレメントで `NORM` + `WIND` を選ぶと、どちらかを持つカードが出ること(実測 1,133件)。APIに `element=NORM&element=WIND` が送られていること
2. **OR**: クラス・タイプ・サブタイプでも同様に複数選択でORが効くこと(サブタイプ `CLERIC`+`SPELL` で 896件)
3. **AND**: サブタイプ `CLERIC` + `SPELL` のANDで **235件相当**(両方を持つカードのみ)が出ること。`CLERIC` だけ/`SPELL` だけのカードが混ざらないこと
4. **AND**: タイプ `REGALIA` + `ITEM` のANDで両方を持つカードのみが出ること
5. **AND**: エレメント `EXALTED` + `WIND` で **15件**、`EXALTED` + `NORM` で **31件** が出ること
6. **エレメントANDのメッセージ**: `FIRE` + `WIND` のAND、`EXALTED` + `CRUX` のAND、3つ以上のANDでメッセージが出ること。**成立する4通り(EXALTED+NORM/FIRE/WATER/WIND)ではメッセージが出ない**こと
7. **単一選択時の互換**: 1つだけ選んだ場合、AND/ORどちらでも従来と同じ結果になること
8. **項目をまたぐ条件**: 異なる項目どうしは従来どおりANDで結合されること(例: クラス=CLERIC かつ エレメント=FIRE)
9. **日本語テキストモードとの整合**: 日本語でカード名・効果を検索した状態でも複数値+AND/ORが正しく効くこと。**同じ絞り込みで英語モードと結果が一致すること**
10. **ページ送り**: AND時に「もっと見る」で重複や取りこぼしが起きないこと。件数表示が実態と矛盾しないこと
11. **スマホ幅(390px)**: 選択UIが操作でき、横スクロールが発生しないこと。選択中の件数が閉じた状態でも分かること
12. **CSP**: 追加UIがstyle属性を使っていないこと(`style-src 'self'`)
13. **サブタイプの表示制御**: 既定で上位20種のみ表示され、「すべて表示」で146種すべてが出ること。選択済みの値が「すべて表示」を閉じても消えないこと
14b. **(2026-07-22追加)PC幅のグループ配置**: 641px以上で4グループが**横に並ぶ**こと(閉じた状態で絞り込み全体が2行に収まること)。**開いたグループだけが全幅に広がる**こと。幅700/900/1180pxで列数が2/3/4に変わり、**いずれも横スクロールが発生しない**こと
14. **開閉の構造**: スマホ幅(≤640px)で**現行どおり「絞り込み・並び替え」で全体が畳まれる**こと。開いた直後は各グループが閉じており、チップが並んでいないこと。PC幅では外側トグルが出ず常時表示であること(現行の挙動が壊れていないこと)
15. **選択状態の可視性**: グループを閉じても選択中の件数バッジが残ること。リセットでバッジが消えること
