## 目的・要望

カード検索の絞り込み条件を**URLに反映し、共有・復元できる**ようにする。

[#19](https://github.com/dozyouneko/ga-card-tools-jp/issues/19) で絞り込みが**複数選択+AND/OR**になり、条件の組み合わせが一気に複雑になった。
「エレメント NORM+WIND(OR)かつ サブタイプ CLERIC+SPELL(AND)」のような検索は口頭やスクリーンショットで
共有しても**相手が手で再現するのが面倒**で、URLで渡せる価値が以前より大きい。

### ユーザー決定(2026-07-23)

| 論点 | 決定 | 理由 |
|---|---|---|
| 「リンクをコピー」ボタン | **置かない** | アドレスバーのURLをそのままコピーして共有する。大会デッキ検索([#14](https://github.com/dozyouneko/ga-card-tools-jp/issues/14))と同じ運用。UI追加ゼロで実装・検証範囲が最小で済む |
| 履歴の扱い | **`replaceState`**(戻るで1手ずつは戻らない) | 大会デッキ検索と同方式。テキスト入力が350msデバウンスのため、`pushState` だと1回の検索で履歴が大量に積まれる |
| 復元時のグループ開閉 | **値のあるグループは開く** | 受け取った人が「何で絞られているか」を一目で確認できる。#19が既定を閉じたのは4グループ全開で一覧が押し出されるためだが、復元で開くのは通常1〜2グループに収まる |

## 画面モック

**なし。** 共有ボタンを置かない決定により、**本対応で新規に増える表示要素はない**(既存のチップ・セレクト・バッジがURLから復元されるだけ)。
復元後の見た目は[#19のモック](../19-カード検索のAND-OR/モック_絞り込みUI.html)の「値を選択した状態」と同一。

## 実測(2026-07-23)

### URLの長さは問題にならない

issueが挙げた懸念「サブタイプを多数選ぶと長くなる」を実測した(`data/translations.js` の `I18N.meta` から算出)。

| 項目 | 選択肢数 | 同名繰り返し | カンマ区切り(URLSearchParams) | カンマ区切り(生文字列) |
|---|---|---|---|---|
| subtype | 147 | 2,193 | 1,317 | 1,025 |
| element | 13 | 177 | 105 | 81 |
| type | 11 | 132 | 102 | 82 |
| class | 9 | 119 | 87 | 71 |

- 上表は**全選択肢を選んだ極端な場合**の文字数。ただし全147種の選択は「絞り込みなし」と等価で実用上あり得ない
- **現実的に重い条件**(サブタイプ10種+エレメント3種)は **206文字**にとどまる
- 本番(Cloudflare Pages)に長いクエリを投げて実測: **500 / 1,000 / 2,000 / 4,000 / 8,000 / 16,000 文字すべて HTTP 200**。
  クエリ長による失敗は起きない

```bash
# 2026-07-23 実測(クエリ長を変えて本番へGET)
for n in 500 1000 2000 4000 8000 16000; do
  qs=$(node -e "process.stdout.write('subtype='+'X'.repeat($n))")
  curl -s -o /dev/null -w "$n → %{http_code}\n" "https://ga-card-tools-jp.pages.dev/?$qs"
done
# 500 → 200 / 1000 → 200 / 2000 → 200 / 4000 → 200 / 8000 → 200 / 16000 → 200
```

### `URLSearchParams` はカンマを `%2C` に符号化する

カンマ区切り案の短縮効果は見かけほど大きくない。

```js
const p = new URLSearchParams(); p.set("a", "X,Y"); p.toString();  // → "a=X%2CY"
```

生文字列を手組みすれば1,025文字まで縮むが、符号化を自前で持つことになり壊れやすい。
**現実的な条件では206文字程度で差が無意味**なため、短縮のために規約を増やす価値はない。

### エキスパンションは添字ではなく prefix を使える

`#f-set` の `value` は `I18N.meta.sets` の**添字**(`set=3`)で、リストの並びが変わると古いURLが別の版を指す。
`sets` の中身を確認したところ、**全55版が `prefixes` を1個だけ持ち、その値は全版で一意**だった。

- したがって `prefixes[0]`(例: `RDO` / `RDOA` / `P22`)を**安定キー**として使える
- 12版は `RDO 1st` のように**空白を含む**が、`URLSearchParams` が `+` に符号化し `get()` で空白に戻るため往復する

## パラメータ仕様

**形式は同名パラメータの繰り返し**とする(`element=NORM&element=WIND`)。

- `buildQuery()` が既にAPIへ同じ形式で送っており(`p.append(param, v)`)、**規約が一つで済む**
- 復元は `p.getAll("element")` でそのまま配列が得られる
- カンマ区切りは前述のとおり短縮効果が薄く、値自体にカンマが現れないという前提も増える

| クエリ名 | 対応するUI | 形式 | 書かない条件(既定値) |
|---|---|---|---|
| `q` | カード名検索 [index.html:53](../../../index.html#L53) | 文字列 | 空 |
| `qtext` | 効果テキスト検索 [index.html:54](../../../index.html#L54) | 文字列 | 空 |
| `element` | エレメント `#g-element` | 繰り返し | 未選択 |
| `class` | クラス `#g-class` | 繰り返し | 未選択 |
| `type` | タイプ `#g-type` | 繰り返し | 未選択 |
| `subtype` | サブタイプ `#g-subtype` | 繰り返し | 未選択 |
| `element_op` / `class_op` / `type_op` / `subtype_op` | 各群のAND/OR | `AND` | **ORのとき(既定)は書かない** |
| `format` | フォーマット `#f-format` | `STANDARD:LEGAL` 形式そのまま | 空 |
| `set` | エキスパンション `#f-set` | `prefixes[0]`(例 `RDO`) | 空 |
| `sort` | 並び替え `#f-sort` | `name` / `cost_memory` / `level` / `power` / `life` / `rarity` | `name`(既定) |
| `order` | 昇順降順 `#order` | `DESC` | **ASCのとき(既定)は書かない** |

方針:

- **クエリ名はAPIのパラメータ名に合わせる**(`class` / `element` / `type` / `subtype`)。`buildQuery()` と対応が取れて読みやすい
- **既定値のパラメータは書かない**。条件なしのときURLは `/` のまま(大会デッキ検索の `saveQuery()` と同じ考え方)
- **値は内部表現のまま大文字で書く**(`AND` / `DESC`)。変換を挟まないぶん取り違えが起きにくい。
  ただし**読み取りは大文字小文字を無視**する(手打ちURLで壊れないように)
- **選択肢に無い値・未知のパラメータは黙って無視する**(大会デッキ検索 `loadQuery()` の
  「選択肢に無い値は無視する(古いURL・手打ちで壊れないように)」と同じ)

### 既存の `?qtext=` 互換

`qtext` は[tools/glossary/app.js:41](../../../tools/glossary/app.js#L41) が
`../../index.html?qtext=<英語キーワード>` の形で生成している**現役のリンク**。

- **`qtext` という名前・意味(効果テキスト欄に入れる)を変えない**
- 他のパラメータが無くても単独で従来どおり動くこと(検証項目11)

## 実装方針

### 書き戻し: `searchCtl` の `onStart(reset=true)` を唯一の起点にする

絞り込みを変える経路は**チップ変更・セレクト変更・並び替え・昇順降順・テキスト入力(デバウンス)・リセット**と多く、
それぞれに書き戻しを足すと漏れる。すべてが最終的に `runSearch(true)` を通るため、
[app.js:78](../../../app.js#L78) の `onStart` で `reset === true` のときに `saveQuery()` を呼ぶ。

- テキスト入力は既に350msデバウンス済みなので、打鍵ごとにURLが書き換わることはない
- `elementAndBlocked()` で検索を止める場合も `onStart` は先に呼ばれる([card-search.js:428](../../../shared/js/card-search.js#L428) が
  431行のブロック判定より前)。**0件が確定する組み合わせもURLに残り、そのまま共有・再現できる**

### `saveQuery()` — ハッシュを明示して保持する

```js
history.replaceState(null, "", location.pathname + (qs ? "?" + qs : "") + location.hash);
```

⚠️ **`location.hash` の連結は必須。** クエリだけを指定して `replaceState` するとハッシュが落ちる(Nodeの `URL` で確認):

```text
new URL("?element=FIRE", "https://x.jp/index.html?element=WIND#card/abc").href
  → "https://x.jp/index.html?element=FIRE"     ← #card/abc が消える
new URL("#card/abc", "https://x.jp/index.html?element=WIND").href
  → "https://x.jp/index.html?element=WIND#card/abc"   ← ハッシュ側は逆にクエリを保つ
```

カード詳細モーダルは開いている間 `#card/<slug>` を持つ([app.js:486](../../../app.js#L486))ため、
連結を忘れるとモーダルを開いた状態で検索し直したときに**共有リンクの対象カードが消える**。

### 復元: `applyUrlQuery()` を全項目対応に拡張する

現在は `qtext` だけを読む[app.js:410-413](../../../app.js#L410-L413)を、上表の全パラメータに広げる。

- **復元中はイベントを発火させない**。チップの `setValues()` / `setMode()` は
  `onChange` を呼ばない実装にする(呼ぶと1項目ごとに検索が走り、無駄なAPIリクエストが並ぶ)
- 復元後に `updateFilterBadge()` を1回呼び、最後に `runSearch(true)` を1回だけ実行する
- **値のあるグループは `g.open = true` にする**(ユーザー決定)。`details` 要素なので直接代入でよく、
  [app.js:112](../../../app.js#L112) の `el.gElement.open = true` に前例がある

### `fillChips()` に復元用APIを追加する

[card-search.js](../../../shared/js/card-search.js) のチップ群は現在 `getValues()` / `getMode()` / `reset()` / `onChange()` / `warnEl`
だけを公開しており、**外から値を入れる手段がない**。次の2つを追加する。

| 追加API | 挙動 |
|---|---|
| `setValues(list)` | `list` に含まれるキーのチェックボックスをONにし、それ以外をOFFにする。`sync()` を呼ぶ。**`onChange` は発火させない** |
| `setMode(mode)` | `"AND"` / `"OR"` を設定して `sync()` を呼ぶ。**`onChange` は発火させない** |

⚠️ **`setValues()` は「すべて表示」に隠れた値を必ず露出させること。**
サブタイプは既定で上位20種だけを出し、残り127種は `restChips`(`hidden = true`)にある
([card-search.js:145-163](../../../shared/js/card-search.js#L145-L163))。
隠れた値を復元するとバッジは「1」なのに**グループを開いてもチップが見当たらない**状態になる。
復元値が `restChips` 側に1つでもあれば `restChips.hidden = false` にし、
`moreBtn` の表記も `− よく使う分だけ表示` に合わせる。

### エキスパンションの添字⇔prefix 変換

`SETS` は [card-search.js:30](../../../shared/js/card-search.js#L30) に閉じているため、変換もそこに置く。

| 追加API | 挙動 |
|---|---|
| `setKeyOf(value)` | `#f-set` の値(添字)から `prefixes[0]` を返す。空・不正なら `""` |
| `setIndexOf(key)` | `prefixes[0]` が `key` に一致する版の添字を文字列で返す。無ければ `""`(=全て) |

`setPrefixes()` が既に「`""` と `Number("")===0` の罠」を扱っている([card-search.js:32-37](../../../shared/js/card-search.js#L32-L37))ので、同じ注意を踏襲する。

### `popstate` で追従する — ただし空振りでAPIを叩かない

大会デッキ検索と同じく `popstate` で `applyUrlQuery()` + `runSearch(true)` を行う。
ただし**直前に自分が書いたクエリと同じなら何もしない**ガードを入れる
(ハッシュ操作などで `popstate` が来たときに無駄な検索を走らせないため)。

## 既存挙動との衝突点(実装時に必ず対処する)

| # | 箇所 | 問題 | 対処 |
|---|---|---|---|
| A | [app.js:524](../../../app.js#L524) `pageshow` | bfcache復帰時に `resetControls(); runSearch(true);` が走り、**URLに条件が残っているのに全てクリアされる**。bfcacheはURLごと復元するため復帰後の表示とURLが食い違う | `resetControls()` の直後に `applyUrlQuery()` と `updateFilterBadge()` を挟む |

## 検討したが採らない案

### localStorage に保存する

前回の絞り込みを次回起動時に復元する案。**共有できないため本issueの目的を満たさない**。
加えて [app.js:432](../../../app.js#L432) のコメント「起動時は必ず『全て』から開始(前回選択の復元を打ち消す)」と正面から衝突する。
**URLに書いてあるときだけ復元する**方式なら、ブラウザのフォーム状態復元を打ち消すという元の意図をそのまま保てる。

### カンマ区切り / `pushState`

前者は前述のとおり短縮効果が薄い。後者はユーザー決定により不採用。

## 実装ステップ

1. **`fillChips()` に `setValues()` / `setMode()` を追加**(`onChange` 非発火。`restChips` の露出を含む)
2. **`setKeyOf()` / `setIndexOf()` を `GA_CARD_SEARCH` に追加**し、エクスポートに載せる
3. **`saveQuery()` を app.js に追加**。`location.hash` の連結を含める
4. **`searchCtl` の `onStart` から `reset === true` のとき `saveQuery()` を呼ぶ**

6. **`pageshow`(bfcache)の復帰処理に `applyUrlQuery()` を挟む**(衝突点A)
7. **`popstate` ハンドラを追加**(同一クエリなら何もしないガード付き)

### 承認した設計からの逸脱 — `popstate` ガードの判定方法

| | 内容 |
|---|---|
| 設計 | 「**直前に自分が書いたクエリ**と同じなら何もしない」(最後の書き込みを変数で覚える) |
| 実装 | 「**`queryString()`(現在の画面から組み立てたクエリ)と `location.search` が同じ**なら何もしない」 |

**実装方式を正とする。** 変数方式は外部要因でURLと変数がずれると
「本当に食い違っているのに追従しない」状態になり得るのに対し、実装方式は
「画面が既にURLどおりなら何もしない」という**状態を持たない判定**で、設計意図
(空振りでAPIを叩かない)をそのまま満たす。`saveQuery()` と組み立て関数を共有するため両者がずれない。

⚠️ この方式は**パラメータの並び順が正規形と違うURLでは文字列一致しない**ため、
履歴移動で余計な再検索が1回走り得る。ただし `reloadFromUrl()` の結果は正しく、
実測でも順序違いURL(`?order=DESC&sort=level&element=WIND`)は正しく復元・正規化された。**実害なし**と判断する。

| 5 | **復元時のグループ開閉**: 値のあるグループが**開いた状態**で表示されること。値のないグループは閉じたままであること |
| 6 | **隠れたサブタイプ**(衝突点C): 上位20種に**入っていない**サブタイプ(例 `SHENJU`)を含むURLを開くと、「すべて表示」が展開済みで**チェック済みチップが見える**こと。バッジの件数と一致すること |
| 7 | **エキスパンション**: `?set=RDO` で該当版が選ばれること。**空白を含む版**(`RDO 1st`)も往復すること。存在しない値(`?set=ZZZ`)は「全て」のまま無視されること |
| 8 | **エレメントANDの0件確定**: `?element=FIRE&element=WIND&element_op=AND` を開くと#19の警告メッセージが出ること。**この状態もURLに残り再現できる**こと |
| 9 | **リセット**(衝突点F): 条件付きURLから「リセット」を押すとURLが `/` に戻り、グループが閉じ、バッジが消えること |
| 10 | **カード詳細との併存**(衝突点B): 絞り込んだ状態でカードを開き `#card/<slug>` が付いた後、**絞り込みを変えてもハッシュが消えない**こと。モーダルを閉じてもクエリが残ること |
| 11 | **既存 `?qtext=` 互換**: [ひとくちキーワード解説](../../../tools/glossary/)のリンクから `?qtext=<英語>` で遷移し、従来どおり効果テキスト欄に入って検索されること |
| 12 | **日本語テキストモード**: `?qtext=追放` のような日本語でもURL往復し、復元後に同じ結果になること |
| 13 | **bfcache復帰**(衝突点A): 条件付きURLから他ページへ移動し**ブラウザバック**すると、絞り込みが**URLどおりに復元**されること(クリアされないこと) |
| 14 | **`popstate`**: 履歴移動でクエリが変わったとき追従すること。**同じクエリのときAPIリクエストが増えない**こと(DevToolsのNetworkで確認) |
| 15 | **不正値の無視**: `?element=NOPE&class=&sort=bogus&unknown=1` を開いても例外を出さず、既定状態で通常表示されること |
| 16 | **URL長**: サブタイプを10種以上選んでも正常に動き、リロードで復元されること |
| 17 | **書き戻し回数**: テキスト検索欄に連続入力しても、URL書き換えが**デバウンス後に1回**であること(打鍵ごとに走らないこと) |
| 18 | **CSP**: 追加コードがstyle属性を使っていないこと(`style-src 'self'`) |

### P4の補足 — `.html` トリムの308はクエリを保持する

解説ページのリンクは `../../index.html?qtext=Materialize` を生成するため、
CLAUDE.mdに記録のある**Cloudflare Pagesの拡張子トリム(308)を必ず通る**。
クエリが落ちないことを実物で確認した。

```text
$ curl -sI "https://ga-card-tools-jp.pages.dev/index.html?qtext=Materialize"
HTTP/2 308
location: /?qtext=Materialize          ← クエリは保持される
（追跡後）HTTP/2 200
```

実ブラウザでも解説ページのリンクをクリック → `/?qtext=Materialize` に着地し、
効果テキスト欄に `Materialize` が入って「50 件を表示 / 全 75 件」を確認した。

**デプロイ反映まで約9分**かかった(pushからCloudflare Pagesの配信更新まで)。
本番確認は反映を待ってから行うこと。
